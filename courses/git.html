Git & GitHub

Git is the most widely used version control system in the world, created by Linus Torvalds in 2005. It tracks changes to files, enables collaboration, and provides a complete history of project evolution.

Beginner
6 Weeks
16 Modules

Overview

Git is a distributed version control system (DVCS) that allows every developer to have a full copy of the entire repository history on their local machine. Unlike centralized systems (SVN, CVS), Git does not require a server for basic operations - commits, diffs, branches, and history are all local operations. This makes Git incredibly fast and allows offline work.

Git works by taking snapshots of files at each commit, storing them as objects in a .git directory. It uses a directed acyclic graph (DAG) to represent history, with each commit pointing to its parent(s). Branches are lightweight pointers to specific commits, making branching and merging fast and cheap.

GitHub, GitLab, and Bitbucket are web-based hosting services that provide remote repositories, pull requests, code review, CI/CD, project management, and social features on top of Git. Mastering Git workflows like Git Flow, GitHub Flow, and trunk-based development is essential for professional software development.

Why Learn Git?

Complete Version History

Track every change ever made to your project. View who changed what, when, and why. Revert to any previous state instantly.

Branching & Experimentation

Create isolated branches for features, bug fixes, or experiments. Merge or discard changes without affecting the main codebase.

Team Collaboration

Multiple developers work on the same codebase simultaneously. Resolve conflicts, review code via pull requests, and coordinate releases.

Backup & Safety Net

With distributed copies, every clone is a full backup. Never lose work again. Git ensures data integrity with SHA-1 hashing.

Notes

1. Git Configuration & Setup

Git stores configuration at three levels: system (all users), global (your user), and local (per repository). User name and email are required for commits. Set up aliases for common commands to save time.

# Configure identity
git config --global user.name "Your Name"
git config --global user.email "email@example.com"

# Set default branch name
git config --global init.defaultBranch main

# Useful aliases
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.st status
git config --global alias.lg "log --oneline --graph --all"

# View configuration
git config --list --global

2. The Three States of Git

Files in Git have three main states: modified (changes not yet staged), staged (marked for commit), and committed (safely stored). The working directory, staging area (index), and repository (HEAD) represent these states.

# Working Directory -> Staging Area
git add file.txt          # Stage a specific file
git add .                 # Stage all changes
git add -p                # Stage interactively (hunk by hunk)

# Staging Area -> Repository
git commit -m "message"   # Commit staged changes
git commit -a -m "msg"    # Stage and commit tracked files

# Check states
git status                # Show current state
git diff                  # Working dir vs staging
git diff --staged         # Staging vs last commit
git diff HEAD             # Working dir vs last commit

3. Branching & Merging

Branches are lightweight pointers to commits. Creating a branch is instantaneous because Git only creates a 41-byte reference file. Merging combines branches, with fast-forward when no divergence exists and 3-way merge when branches have diverged.

# Branch operations
git branch feature-x       # Create branch
git switch feature-x       # Switch to branch
git checkout -b feature-x  # Create and switch

# Merge feature-x into main
git switch main
git merge feature-x

# If conflict occurs, resolve then:
git add resolved-file.txt
git commit                # Complete merge commit

# List branches
git branch -a              # All branches (local + remote)
git branch -d feature-x    # Delete merged branch

4. Working with Remote Repositories

Remotes are URLs to remote copies of your repository. Origin is the default name for your forked/cloned remote. Upstream is often the original repository you forked from. Fetch gets changes without merging, pull fetches and merges.

# Remote management
git remote add origin https://github.com/user/repo.git
git remote -v              # List remotes
git remote remove upstream # Remove a remote

# Syncing
git fetch origin           # Get changes, don't merge
git pull origin main       # Fetch + merge
git push origin main       # Push to remote

# Push to upstream and set tracking
git push -u origin main    # -u sets upstream tracking

# Pull with rebase instead of merge
git pull --rebase origin main

5. Interactive Rebase

Interactive rebase (git rebase -i) allows rewriting commit history by squashing, rewording, dropping, reordering, or editing commits. Use it to clean up local history before pushing. Never rebase commits that others have based work on.

# Rebase last 3 commits interactively
git rebase -i HEAD~3

# In the editor:
# pick - use commit as-is
# reword - change commit message
# squash - combine with previous commit
# fixup - squash but discard message
# drop - remove commit
# edit - stop to amend

# Abort if things go wrong
git rebase --abort

# Continue after resolving conflicts
git rebase --continue

6. Git Revert vs Reset

git revert creates a new commit that undoes changes - safe for shared history. git reset moves the branch pointer and optionally modifies the staging area and working directory. Use reset only for local/private branches.

# Safe undo for shared history
git revert HEAD            # Revert last commit
git revert abc1234         # Revert specific commit

# Reset modes (DANGER: rewrites history)
git reset --soft HEAD~1    # Move pointer, keep changes staged
git reset --mixed HEAD~1   # Move pointer, keep working changes
git reset --hard HEAD~1    # Move pointer, DISCARD all changes

# Recover from hard reset
git reflog                 # Find lost commit hash
git reset --hard abc1234   # Restore to that state

7. Git Hooks with Husky

Git hooks are scripts that run automatically before or after Git events like commit, push, or merge. Husky is a popular Node.js tool that makes managing hooks easier. Use hooks to enforce code quality and prevent bad commits.

// .husky/pre-commit
// Run linter and tests before commit
npx lint-staged
npm test

// .husky/commit-msg
// Enforce conventional commit format
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"
npx --no-install commitlint --edit "$1"

// .husky/pre-push
// Run full test suite before push
npm run build
npm run test:ci

// package.json with lint-staged
{
  "lint-staged": {
    "*.{js,ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.css": ["stylelint --fix"]
  }
}

8. GitHub Pull Requests & Collaboration

Pull requests (PRs) are the core of GitHub collaboration. They propose changes from one branch to another, enabling code review, discussion, and automated checks before merging. A good PR is focused, well-described, and includes test coverage.

# PR workflow with GitHub CLI
gh pr create --title "Add user authentication" \
  --body "Implements JWT-based auth with refresh tokens" \
  --base main --head feature/auth

gh pr review --approve
gh pr merge --squash

# PR best practices:
# 1. Small, focused changes (under 400 lines)
# 2. Descriptive title and description
# 3. Link to related issues
# 4. Include screenshots for UI changes
# 5. Add tests for new functionality
# 6. Keep commit history clean

Examples

Example 1: Feature Branch Workflow

# Start from updated main
git checkout main
git pull origin main

# Create feature branch
git checkout -b feature/user-profile

# Work and commit
echo "profile code" > profile.js
git add profile.js
git commit -m "feat: add user profile component"

# Keep up with main during development
git fetch origin
git rebase origin/main

# Push and create PR
git push -u origin feature/user-profile
gh pr create --title "feat: user profile" --body "Implements user profile page with avatar, bio, and settings"

# After PR approval and merge, delete branch
git checkout main
git pull origin main
git branch -d feature/user-profile

Example 2: Resolving Merge Conflicts

# When git merge produces a conflict:
git merge feature/login
# CONFLICT in src/auth.js

# View conflicting file
cat src/auth.js
# <<<<<<< HEAD
# const token = localStorage.getItem('token');
# =======
# const token = sessionStorage.getItem('token');
# >>>>>>> feature/login

# Fix by editing the file - choose one or combine:
const token = localStorage.getItem('token') || sessionStorage.getItem('token');

# Mark as resolved and complete merge
git add src/auth.js
git commit

# Or abort the merge entirely
git merge --abort

# Use a merge tool
git mergetool  # Opens configured tool (vscode, meld, kdiff3)
