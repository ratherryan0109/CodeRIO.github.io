Project-Based Learning Explained

Think of it like this:

So far, you have been learning individual HTML concepts like learning the individual instruments in an orchestra. You know how the violin sounds, how the drums work, and what the trumpet does. Now it is time to play a complete symphony — combining everything you have learned into a real, working project that does something meaningful. That is what this module is about.

This capstone module is different from the previous ones. Instead of learning new HTML concepts, you will apply everything you have learned so far to build real projects. There is no better way to learn HTML than by building actual websites. You will make mistakes (and learn from them), you will look things up (and remember them better), and you will end up with a portfolio of work you can show employers.

We will build two major projects together: a Personal Portfolio Page and a Product Landing Page. After that, you will have a list of additional project ideas to build on your own. Remember: you do not need CSS or JavaScript to create functional, well-structured pages — focus on getting the HTML right first. Each project reinforces specific skills from the previous modules.

Project 1: Personal Portfolio

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Jane Dev | Front-End Developer</title>
  <meta name="description" content="Portfolio of Jane Dev, a front-end developer specializing in accessible, responsive web design.">
</head>
<body>
  <header>
    <h1>Jane Dev</h1>
    <nav aria-label="Main">
      <a href="#about">About</a>
      <a href="#skills">Skills</a>
      <a href="#projects">Projects</a>
      <a href="#contact">Contact</a>
    </nav>
  </header>

  <main>
    <section id="about">
      <h2>About Me</h2>
      <figure>
        <img src="profile.jpg" alt="Jane Dev smiling at her desk">
        <figcaption>Building the web, one tag at a time</figcaption>
      </figure>
      <p>I am a front-end developer with a passion for clean, accessible HTML...</p>
    </section>

    <section id="skills">
      <h2>Skills</h2>
      <ul>
        <li>HTML5 & Semantic Markup</li>
        <li>CSS3 & Flexbox/Grid</li>
        <li>JavaScript (ES6+)</li>
      </ul>
    </section>

    <section id="projects">
      <h2>Projects</h2>
      <article>
        <h3>Weather App</h3>
        <p>Real-time weather data using OpenWeather API...</p>
        <a href="#">Live Demo</a> | <a href="#">Source Code</a>
      </article>
    </section>

    <section id="contact">
      <h2>Contact</h2>
      <form action="/contact" method="POST">
        <label for="name">Name:</label>
        <input type="text" id="name" name="name" required>
        <label for="email">Email:</label>
        <input type="email" id="email" name="email" required>
        <label for="msg">Message:</label>
        <textarea id="msg" name="msg" rows="5"></textarea>
        <button type="submit">Send</button>
      </form>
    </section>
  </main>

  <footer>
    <p>&copy; 2026 Jane Dev. All rights reserved.</p>
  </footer>
</body>
</html>

Key Concepts Used:

  • Structure: header, nav, main, section, article, footer — the full semantic layout from Module 11.
  • Navigation: Bookmark links (#about, #skills) for single-page navigation from Module 6.
  • Form: A contact form with labels, inputs, and textarea from Module 10.
  • Figure: Image with caption for the profile photo from Module 7.

Key Concepts Used:

  • Pricing Table: Uses semantic <article> for each plan card with <ul> for features from Module 8.

  • Links: Each plan has a call-to-action link, demonstrating link behavior from Module 6.
  • Headings: Proper hierarchy — h2 for the section, h3 for each plan — from Module 4.

Advanced Project Ideas

Blog Homepage

Restaurant Website

Build a site with a gallery (images with figures), a menu (description list), location/contact (table with hours), and a reservation form.

Online Store

Create a product listing with a table of product comparisons, a shopping cart page with a table, and a checkout form with all input types.

Event Page

Build an event landing page with a hero section, schedule (table with colspan/rowspan), speaker cards (figures), FAQ (description list), and registration form.

Recipe Book

Personal Dashboard

Capstone Assignment

Build a Complete Portfolio Website! This is your final HTML capstone project. Create a multi-page portfolio website with at least 3 pages linked together:

  • index.html — Home page with hero section, about blurb, featured projects grid (using article elements), and skills list
  • projects.html — Detailed projects page with a table of projects (name, tech used, year, link) and individual project articles with images
  • contact.html — Contact page with a complete form (name, email, subject dropdown, message textarea, preferences checkboxes), a table with office hours, and address information using the address element

All pages must share the same header/nav and footer (you can copy them). Every page must have: proper DOCTYPE, lang attribute, unique title and meta description, semantic HTML throughout, at least one image with alt text, proper heading hierarchy, and a footer with copyright. Add ARIA labels where helpful and include Open Graph meta tags for social sharing.

Hints:

  • Start by sketching your layout on paper before writing any code. Plan which semantic elements go where.
  • Use the <time> element for dates and <address> for contact information.
  • Validate every page with the W3C Nu HTML Checker before calling it complete.
  • Share your finished portfolio with friends or on social media — you earned it!

Interview Questions

Q: Walk me through how you would structure a portfolio website in HTML.

I would start with a header containing the site name and navigation. The main content would use sections for About, Skills, Projects (each project as an article), and Contact. I would use semantic HTML throughout — nav for navigation, main for primary content, aside for a sidebar if needed, and footer for copyright. Each page would have a unique title and meta description, proper heading hierarchy (one h1, followed by h2s for sections), and all images would have descriptive alt text.

Q: How would you handle navigation between multiple pages without CSS or JavaScript?

I would use relative links in an <a> tag within a <nav> element. For example, <a href="about.html">About</a>. The nav should be inside a <header> and marked with aria-label="Main navigation". For the current page, I could add aria-current="page" to the active link to indicate it to screen readers.

Q: What would you include in the head section of a real-world project?

Essential tags: charset, viewport for mobile, title (unique per page), meta description, Open Graph tags for social sharing, Twitter Card tags, canonical URL (to prevent duplicate content issues), favicon link, and any preconnect/preload hints for performance. I would also include structured data (JSON-LD) for articles or organization when appropriate.

Common Mistakes

  • Trying to code first, plan later: Rushing into code without planning the page structure leads to messy HTML. Sketch your layout first.
  • Copy-pasting without understanding: When building projects, it is tempting to copy code. Type everything manually to build muscle memory.
  • Ignoring validation: Always run your HTML through the W3C Nu HTML Checker. It catches errors you would miss.
  • Not testing in multiple browsers: Your page might look fine in Chrome but broken in Firefox. Test in at least 2-3 browsers.
