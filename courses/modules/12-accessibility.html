Web Accessibility

Beginner 50 min Module 12 of 14

Learning Objectives

  • Understand what web accessibility (a11y) means and why it matters
  • Use ARIA attributes to enhance accessibility when needed
  • Write effective alt text for images and meaningful link text
  • Ensure keyboard navigability and proper color contrast

Accessibility Explained

Think of it like this:

Building a website without accessibility is like building a library with steps but no ramp. Some people simply cannot enter. Web accessibility (often written "a11y" — 11 letters between A and Y) ensures that everyone, regardless of ability, can use your website. This includes people who are blind or visually impaired, deaf or hard of hearing, have motor disabilities and cannot use a mouse, or have cognitive disabilities.

Accessibility is not an afterthought — it should be built into your HTML from the start. The foundation of accessibility is semantic HTML. Using the correct heading hierarchy, proper form labels, meaningful link text, and appropriate elements (like <button> for actions and <nav> for navigation) goes a long way. When semantic HTML is not enough, you can use ARIA (Accessible Rich Internet Applications) attributes to provide additional information to assistive technologies.

The Web Content Accessibility Guidelines (WCAG) define four principles: content must be Perceivable (available to the senses), Operable (usable with various input methods), Understandable (clear and predictable), and Robust (works with current and future technologies). These four principles (POUR) are the foundation of modern web accessibility and are enforced by law in many countries.

Beginner Example

<!-- Good alt text -->
<img src="dog.jpg" alt="A golden retriever puppy playing in a park">

<!-- Decorative image -->
<img src="divider.png" alt="">

<!-- Meaningful link text -->
<a href="pricing.html">View our pricing plans</a>

<!-- Bad: vague link text -->
<a href="pricing.html">Click here</a>

Line-by-Line Explanation:

  • Line 2: Descriptive alt text — describes both the subject (golden retriever puppy) and the context (playing in a park). Screen readers will read this aloud.
  • Line 5: alt="" — empty alt text for a decorative image. This tells screen readers to skip it entirely, which is better than reading "divider.png" or a useless description.
  • Lines 8, 11: Compare "View our pricing plans" (descriptive, tells you what the link does) with "Click here" (vague, requires reading surrounding text to understand).

Intermediate Example

<!-- ARIA labels for clarity -->
<nav aria-label="Main navigation">
  <ul>
    <li><a href="/">Home</a></li>
    <li><a href="/shop">Shop</a></li>
  </ul>
</nav>

<!-- Keyboard-accessible button -->
<button aria-label="Close dialog" onclick="closeDialog()">
  <i class="fas fa-times"></i>
</button>

<!-- Form validation with aria -->
<label for="email">Email:</label>
<input type="email" id="email" name="email" required
       aria-describedby="email-hint">
<p id="email-hint">We will never share your email.</p>

Line-by-Line Explanation:

  • Lines 2-8: aria-label="Main navigation" gives this nav a descriptive label. If there are multiple navs, screen reader users can distinguish them.
  • Lines 11-13: A button with only an icon. The aria-label provides a text alternative — screen readers will say "Close dialog" instead of ignoring the icon.
  • Lines 17-18: aria-describedby associates a hint paragraph with the input. Screen readers announce the hint when the input is focused.

Advanced Example

<!-- Accessible accordion -->
<div class="accordion">
  <button aria-expanded="false" aria-controls="panel1"
          id="accordion-btn1">
    Section 1
  </button>
  <div id="panel1" role="region" aria-labelledby="accordion-btn1" hidden>
    <p>Content for section 1...</p>
  </div>
</div>

<!-- Skip navigation link -->
<a href="#main-content" class="skip-link">Skip to main content</a>

<!-- Accessible form with error handling -->
<label for="username">Username:</label>
<input type="text" id="username" name="username"
       aria-required="true"
       aria-invalid="true"
       aria-errormessage="username-error">
<p id="username-error" role="alert">Username is required.</p>

<!-- Live region for dynamic content -->
<div aria-live="polite" aria-atomic="true">
  Cart: 3 items
</div>

Line-by-Line Explanation:

  • Lines 3-10: An accessible accordion. aria-expanded tells screen readers whether the panel is open. aria-controls links the button to the panel it controls. aria-labelledby gives the panel an accessible name from the button.
  • Line 12: A "Skip to main content" link — the first focusable element on the page. It lets keyboard users skip repetitive navigation links.
  • Lines 21-22: aria-invalid="true" marks the input as having an error. aria-errormessage points to the error message element.
  • Line 25: aria-live="polite" — a "live region" that announces dynamic updates (like cart count changes) to screen readers without interrupting the current task.

Assignment

Take an existing HTML page (like the registration form from Module 10) and audit it for accessibility. Fix all issues you find. Add: a skip-to-content link, proper alt text on all images, aria-labels on interactive elements where needed, aria-describedby hints on form fields, role attributes on dynamic elements, and ensure all interactive elements are keyboard-accessible. Run a Lighthouse accessibility audit and aim for a score of at least 90. Save the fixed version as accessible-form.html.

Hints:

  • Use the Chrome DevTools Lighthouse tab (Accessibility audit) to get a baseline score and specific recommendations.
  • Tab through your page — can you reach every interactive element? Do you see a visible focus ring?
  • Install the WAVE browser extension for real-time accessibility feedback.

Interview Questions

Q: What is the difference between aria-label and aria-labelledby?

Both provide an accessible name. aria-label accepts a string directly (e.g., aria-label="Close"). aria-labelledby references the id of another element that contains the label text (e.g., aria-labelledby="heading-id"). Use aria-labelledby when the visible text on the page is the best label.

Q: What is the "first rule of ARIA"?

Do not use ARIA if you can use a native HTML element. Native HTML elements have built-in semantic meaning, keyboard handling, and focus management. For example, use <button> instead of <div role="button">. ARIA should only supplement native semantics when they are insufficient.

Q: What is the WCAG contrast ratio requirement for normal-sized text?

WCAG AA requires a contrast ratio of at least 4.5:1 for normal-sized text (under 18px / 14px bold) and 3:1 for large text (over 18px or 14px bold). This ensures readability for users with low vision or color blindness. You can check contrast with tools like WebAIM's Contrast Checker.

Common Mistakes

  • Using "click here" as link text: Screen reader users often navigate by listing links. "Click here" tells them nothing about where the link goes.
  • Missing focus indicators: Removing outline: none from focused elements without providing an alternative. Keyboard users cannot see where they are on your page.
  • Overusing ARIA: Adding ARIA to everything when native HTML handles it. This can actually make things less accessible if misused.
  • Poor color contrast: Light gray text on white backgrounds is hard to read for everyone, not just people with visual impairments.
