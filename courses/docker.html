Docker

Docker is a platform for developing, shipping, and running applications in containers. Containers are lightweight, portable, and consistent environments that package application code with all dependencies, eliminating the "it works on my machine" problem.

Advanced
6 Weeks
16 Modules

Overview

Docker was launched in 2013 by Solomon Hykes as an internal project at dotCloud. It popularized containerization by making it easy to create, deploy, and run containers using a simple CLI and a daemon-based architecture. Unlike virtual machines, containers share the host OS kernel, making them more lightweight and faster to start.

Docker uses images (read-only templates) to create containers. Images are built from Dockerfiles, a declarative syntax for specifying the environment, dependencies, and commands. Docker Hub is the default public registry for sharing images. Docker Compose defines multi-container applications using a YAML configuration file.

Docker has revolutionized software development and operations. It ensures consistency across development, testing, and production environments. Docker is the foundation for modern microservices, CI/CD pipelines, cloud-native applications, and container orchestration platforms like Kubernetes, Docker Swarm, and Amazon ECS.

Why Learn Docker?

Consistent Environments

Eliminate "it works on my machine" problems. Containers ensure your app runs identically on your laptop, CI server, and production.

Fast Deployment & Scaling

Containers start in seconds, not minutes. Scale horizontally by spinning up more containers. Rollback instantly to previous versions.

Lightweight Resource Usage

Unlike VMs, containers share the host kernel and require no guest OS. Run hundreds of containers on a single host with minimal overhead.

Industry Standard

Docker is used by 80%+ of organizations. Required knowledge for DevOps, SRE, backend, and cloud engineering roles.

Notes

1. Essential Docker Commands

Docker CLI follows the pattern: docker [object] [action]. Common objects are container, image, volume, network, and system. Use --help flag for detailed documentation.

# Container lifecycle
docker run -d -p 8080:80 --name myapp nginx
docker stop myapp
docker start myapp
docker rm -f myapp

# Inspecting
docker ps -a                 # List all containers
docker logs -f myapp         # Follow logs
docker exec -it myapp bash   # Enter running container
docker inspect myapp         # Detailed metadata
docker stats                 # Live resource usage

# Images
docker pull ubuntu:22.04
docker build -t myapp:1.0 .
docker images
docker rmi myapp:1.0
docker system prune -a       # Clean everything

2. Writing Efficient Dockerfiles

Each Dockerfile instruction creates a layer. Minimize layers, order instructions from least to most frequently changing to maximize cache usage. Use multi-stage builds to keep final images small.

# Efficient Node.js Dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER nodejs
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1
CMD ["node", "dist/index.js"]

3. Docker Compose Multi-Container Setup

Docker Compose defines multi-container applications in YAML. Each service runs in its own container with isolated environments. Networks enable service discovery by service name.

# docker-compose.yml
version: '3.8'

services:
  api:
    build: ./api
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/app
      - REDIS_URL=redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    volumes:
      - ./api:/app
      - /app/node_modules

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: app
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d app"]
      interval: 5s

  cache:
    image: redis:7-alpine
    volumes:
      - redis_data:/data

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - api

volumes:
  pgdata:
  redis_data:

4. Docker Networking

Docker provides several network drivers. Bridge is the default for standalone containers. Overlay enables multi-host communication in Swarm. Host gives a container direct access to the host network. Macvlan assigns MAC addresses to containers.

# Create custom networks
docker network create --driver bridge my-net
docker network create --driver overlay --attachable swarm-net

# Run containers on specific networks
docker run -d --name app --network my-net myapp
docker run -d --name db --network my-net postgres

# Connect/disconnect containers
docker network connect my-net container1
docker network disconnect my-net container1

# Internal DNS resolution - containers can reach each other by name
# app can reach db at hostname: db, port: 5432

# List and inspect networks
docker network ls
docker network inspect my-net

5. Multi-Stage Builds

Multi-stage builds use multiple FROM statements to create intermediate build stages. Each stage can use different base images. Only the final stage's layers are kept in the final image, enabling tiny production images.

# Build stage: full SDK for compilation
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["MyApp.csproj", "."]
RUN dotnet restore
COPY . .
RUN dotnet publish -c Release -o /app

# Runtime stage: minimal runtime image
FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine
WORKDIR /app
COPY --from=build /app .
EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080
ENTRYPOINT ["dotnet", "MyApp.dll"]

# Check image sizes
docker images
# SDK image: ~1.8GB
# Final image: ~200MB

6. Volumes & Data Management

Docker volumes persist data independently of container lifecycle. Named volumes are managed by Docker, bind mounts map host directories, and tmpfs mounts store data in memory. Use volumes for databases, user uploads, and configuration.

# Create and use named volumes
docker volume create app-data
docker run -v app-data:/data myapp

# Bind mount (development)
docker run -v $(pwd):/app -w /app node:20 npm run dev

# Backup a volume
docker run --rm -v app-data:/source -v $(pwd):/backup \
  alpine tar czf /backup/backup.tar.gz -C /source .

# Restore a volume
docker run --rm -v app-data:/target -v $(pwd):/backup \
  alpine tar xzf /backup/backup.tar.gz -C /target

# Volume drivers for cloud storage
docker volume create --driver=rexray/ebs my-ebs-volume

7. Docker Security Best Practices

Security in Docker is multi-layered: image security (scanning for vulnerabilities, using minimal base images), runtime security (read-only root, drop capabilities, non-root user), and host security (networking, resource limits).

# Dockerfile security directives
FROM node:20-alpine
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser
COPY --chown=appuser:appgroup . /app
RUN chmod -R 755 /app

# Runtime security
docker run \
  --read-only \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --security-opt=no-new-privileges:true \
  --security-opt=seccomp=default.json \
  -m 512m \
  --cpus=1 \
  myapp

# Image scanning
docker scout quickview myapp:latest
docker scout recommendations myapp:latest

8. Docker Swarm & Orchestration

Docker Swarm turns a group of Docker hosts into a single virtual host. It provides native clustering, service discovery, load balancing, scaling, and rolling updates. Services are the Swarm equivalent of containers.

# Initialize Swarm
docker swarm init --advertise-addr 192.168.1.10
docker swarm join-token worker

# Deploy a service
docker service create \
  --name web \
  --replicas 3 \
  --publish 80:80 \
  --update-delay 10s \
  --rollback-monitor 20s \
  nginx:alpine

# Scale the service
docker service scale web=5

# Rollback if needed
docker service update --rollback web

# Stack deploy from compose file
docker stack deploy -c docker-compose.yml myapp

# Manage nodes
docker node ls
docker node promote node2
docker node demote node2

Example 1: Full-Stack App with Docker Compose

version: '3.8'
services:
  frontend:
    build:
      context: ./frontend
      target: production
    ports:
      - "3000:3000"
    environment:
      - NEXT_PUBLIC_API_URL=http://api:4000

Example 2: CI/CD Pipeline with Docker

# .github/workflows/deploy.yml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            myapp:latest
            myapp:${{ github.sha }}

      - name: Deploy to server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.HOST }}
          script: |
            docker pull myapp:latest
            docker stack deploy -c docker-compose.yml myapp
            docker image prune -f
