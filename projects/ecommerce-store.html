E-Commerce Store

Intermediate 8-12 hours React, Context API, Stripe, Node.js

A comprehensive engineering guide to building a production-ready e-commerce storefront with React components, state management, Stripe payment integration, and a complete checkout flow.

Project Overview

This e-commerce store project simulates a real production environment where you build a complete online storefront from scratch using React. You will implement a product catalog with categories and filters, a shopping cart with full CRUD operations, a multi-step checkout process, and Stripe payment integration. The project covers state management with React Context and useReducer, API integration with a mock backend (or Node.js/Express server), responsive product grids, image galleries, and order management. The architecture follows industry patterns used by companies like Shopify, BigCommerce, and Magento.

Problem Statement

Building an e-commerce site involves complex state management (cart, user, products), secure payment processing, responsive product displays, and a smooth checkout flow. Many developers rely on platforms like Shopify or WooCommerce but don't understand the underlying architecture. The challenge is building a custom e-commerce solution that handles: concurrent cart updates, inventory management, tax calculations, shipping logic, payment gateway integration, order confirmation, and error recovery - all while providing a seamless user experience.

Real World Use Cases

  • Small Business Storefront: A local boutique sells handmade products online with categories, size/color variants, and local pickup options.
  • Digital Products Marketplace: A creator sells digital downloads (ebooks, templates, software) with instant delivery after payment and license key generation.
  • Multi-vendor Platform: A marketplace connects multiple vendors with customers, each vendor managing their own inventory, pricing, and shipping profiles.

Industry Relevance

Global e-commerce sales exceeded $5.7 trillion in 2025 and continue growing. Companies like Shopify, BigCommerce, and WooCommerce power millions of stores. Understanding e-commerce architecture is valuable for roles at any tech company that sells products online. React is used by 40%+ of professional developers and is the foundation for modern e-commerce frontends (Shopify Hydrogen, Next.js Commerce). Stripe processes payments for millions of businesses worldwide and has the most developer-friendly API in the payments industry.

Architecture Overview

The application follows a microservices-inspired architecture with a React frontend and a Node.js/Express backend. The frontend uses React Router for client-side routing with pages for Home, Products, Product Detail, Cart, Checkout, and Order Confirmation. State management is split into contexts: CartContext (items, totals, quantity), AuthContext (user, token, login/logout), and UIContext (modals, notifications, loading states). The backend exposes RESTful endpoints: /api/products (CRUD with filtering), /api/cart (sync), /api/orders (create, history), /api/checkout (payment intent), and /api/webhooks/stripe (payment confirmation). Stripe Elements securely collects payment details without sensitive data touching the server. Product images are served from a CDN (Cloudinary or similar) with responsive srcset for different screen sizes.

Step 1: Cart Context with useReducer

Implement a robust cart state management system using React Context and useReducer. The reducer handles all cart operations with immutability and supports both client-side and server-synced cart states.

const CartContext = createContext();

const initialState = {
  items: [],
  shippingAddress: null,
  paymentMethod: null,
  couponCode: null,
  discount: 0,
  loading: false,
  error: null
};

function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM': {
      const existingIndex = state.items.findIndex(
        i => i.id === action.payload.id &&
        i.variant === action.payload.variant
      );
      if (existingIndex >= 0) {
        const updatedItems = [...state.items];
        updatedItems[existingIndex] = {
          ...updatedItems[existingIndex],
          quantity: updatedItems[existingIndex].quantity + action.payload.quantity
        };
        return { ...state, items: updatedItems };
      }
      return { ...state, items: [...state.items, { ...action.payload }] };
    }
    case 'REMOVE_ITEM':
      return { ...state, items: state.items.filter(i => i.id !== action.payload) };
    case 'UPDATE_QUANTITY': {
      const { id, quantity } = action.payload;
      if (quantity <= 0) {
        return { ...state, items: state.items.filter(i => i.id !== id) };
      }
      return {
        ...state,
        items: state.items.map(i => i.id === id ? { ...i, quantity } : i)
      };
    }
    case 'CLEAR_CART':
      return { ...initialState };
    case 'APPLY_COUPON':
      return { ...state, couponCode: action.payload.code, discount: action.payload.discount };
    case 'SET_SHIPPING':
      return { ...state, shippingAddress: action.payload };
    case 'SET_PAYMENT':
      return { ...state, paymentMethod: action.payload };
    case 'SET_LOADING':
      return { ...state, loading: action.payload };
    case 'SET_ERROR':
      return { ...state, error: action.payload };
    default:
      return state;
  }
}

Step 3: Product Filtering with URL Params

Implement a URL-based filtering system where filter state is stored in URL query parameters, enabling shareable filtered links and browser back/forward navigation.

import { useSearchParams } from 'react-router-dom';

function useProductFilters(products) {
  const [searchParams, setSearchParams] = useSearchParams();

  const filters = {
    category: searchParams.get('category') || 'all',
    minPrice: Number(searchParams.get('minPrice')) || 0,
    maxPrice: Number(searchParams.get('maxPrice')) || Infinity,
    rating: Number(searchParams.get('rating')) || 0,
    sort: searchParams.get('sort') || 'newest',
    search: searchParams.get('search') || '',
    inStock: searchParams.get('inStock') === 'true'
  };

  const setFilter = (key, value) => {
    setSearchParams(prev => {
      if (value === null || value === '' || value === 'all' || value === 0) {
        prev.delete(key);
      } else {
        prev.set(key, value);
      }
      return prev;
    });
  };

  const clearFilters = () => setSearchParams({});

  const filtered = useMemo(() => {
    let result = [...products];
    if (filters.category !== 'all') {
      result = result.filter(p => p.category === filters.category);
    }
    result = result.filter(p => p.price >= filters.minPrice && p.price <= filters.maxPrice);
    if (filters.rating > 0) {
      result = result.filter(p => p.rating >= filters.rating);
    }
    if (filters.search) {
      const q = filters.search.toLowerCase();
      result = result.filter(p =>
        p.name.toLowerCase().includes(q) ||
        p.description.toLowerCase().includes(q) ||
        p.tags.some(t => t.toLowerCase().includes(q))
      );
    }
    if (filters.inStock) {
      result = result.filter(p => p.stock > 0);
    }
    switch (filters.sort) {
      case 'price-asc': result.sort((a, b) => a.price - b.price); break;
      case 'price-desc': result.sort((a, b) => b.price - a.price); break;
      case 'rating': result.sort((a, b) => b.rating - a.rating); break;
      case 'name': result.sort((a, b) => a.name.localeCompare(b.name)); break;
      default: result.sort((a, b) => new Date(b.createdAt) - new Date(a.createdAt)); break;
    }
    return result;
  }, [products, filters]);

  const categories = useMemo(() =>
    [...new Set(products.map(p => p.category))],
    [products]
  );

  return { filters, setFilter, clearFilters, filtered, categories };
}

Deployment Guide

Backend: Deploy the Node.js server to Heroku, Railway, Render, or AWS Elastic Beanstalk. Use PM2 for process management. Set up a MongoDB Atlas cluster or PostgreSQL on RDS.

Stripe Webhooks: Register your production webhook endpoint in the Stripe Dashboard. Listen for payment_intent.succeeded, payment_intent.payment_failed, and checkout.session.completed events. Use the Stripe CLI for local webhook testing.

Domain & SSL: Set up a custom domain with your DNS provider. Configure SSL certificates through your hosting provider (automatic on Vercel/Netlify). Set up a CDN for product images (Cloudinary, imgix, or CloudFront).

Common Mistakes

1. Insecure Payment Data Handling
Never send raw credit card numbers to your server. Stripe Elements encrypts card data client-side and returns a token/payment method ID. Your server only receives this token. PCI DSS compliance requires that card data never touches your server. Always use Stripe's recommended integration pattern and never log payment details.
2. Cart State Desync Between Client and Server
When a user adds items to cart on one device, the cart isn't synced to their account. Implement cart merging on login: the server-side cart and client-side cart should merge, preferring server items for products that exist in both. Use the server's cart as the source of truth once authenticated.
3. Missing Inventory Checks During Checkout
A user adds 10 items to cart but only 3 are in stock. Without real-time inventory checks, the order will fail after payment. Implement atomic inventory checks during checkout: lock inventory when the checkout process starts, release on timeout if not completed, and decrement on successful payment. Use database transactions for stock operations.
4. Not Handling Payment Webhook Idempotency
Stripe webhooks can be delivered multiple times (retry logic). Without idempotency, the same order can be processed twice. Store processed webhook event IDs in a database with a unique constraint. Check for duplicate events before processing. Use the Stripe-Signature header to verify webhook authenticity.

Interview Questions

Q: Explain the difference between Context API and Redux. When would you use each?
A: Context API is built into React and is suitable for small to medium applications with simple state (theme, auth, locale). Redux provides a more structured approach with middleware, devtools, time-travel debugging, and normalized state. Use Context for low-frequency updates and simple state. Use Redux for complex state with frequent updates, cross-cutting concerns, or when you need middleware (sagas, thunks). For e-commerce, Context + useReducer works well for cart state, but a global auth store might benefit from Redux if you need complex permission logic.
Q: How would you optimize an e-commerce product listing page with 10,000+ products?
A: Implement server-side pagination (20-50 products per page) with cursor-based or offset pagination. Use virtual scrolling (react-window or react-virtuoso) for the product grid. Implement server-side filtering and sorting to avoid loading all products. Use GraphQL to fetch only the fields needed (product name, price, image URL). Lazy load product images with Intersection Observer. Cache API responses with React Query or SWR. Prefetch the next page on hover/scroll.
Q: Describe the Stripe payment flow from checkout to order confirmation.
A: (1) Client collects payment details via Stripe Elements (CardElement). (2) Client sends the total amount to the server. (3) Server creates a PaymentIntent with the amount, currency, and payment_method_types. (4) Server returns the client_secret to the client. (5) Client calls stripe.confirmCardPayment(client_secret) which handles 3D Secure if needed. (6) Stripe processes the payment and returns a result. (7) Server listens for payment_intent.succeeded webhook and updates the order status. (8) Client redirects to the order confirmation page with the payment status.
Q: What is the difference between controlled and uncontrolled components in React?
A: Controlled components have their state managed by React via useState/useReducer and onChange handlers. The input's value is always driven by React state. Uncontrolled components manage their own DOM state internally, accessed via refs. Controlled components give more control (instant validation, conditional disabling, format on input) but require more code. Uncontrolled are simpler but less flexible. In e-commerce, use controlled components for the checkout form (validation required) and uncontrolled for a search input (less critical).
Q: How do you implement optimistic updates in a cart system?
A: Optimistic updates immediately apply changes to the UI before the server confirms, creating a faster user experience. Pattern: (1) Update local cart state immediately with the new item. (2) Send the API request to sync with the server. (3) If successful, keep the optimistic state. (4) If failed, rollback to the previous state and show an error toast. Use a snapshot of the previous state for rollback. Example: when adding to cart, immediately increment the cart badge count and add the item to the local cart, then POST to /api/cart/add. On error, revert to the snapshot.
