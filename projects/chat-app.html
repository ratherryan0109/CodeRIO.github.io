Chat Application

Intermediate 8-12 hours Socket.io, Node.js, Express, MongoDB

A comprehensive engineering guide to building a real-time chat application with Socket.io, room management, user authentication, typing indicators, message formatting, and online presence tracking.

Project Overview

This real-time chat application project teaches you how to build a complete messaging platform from scratch using Socket.io for bidirectional real-time communication, Node.js/Express for the backend, and MongoDB for persistent message storage. Users can create accounts, join multiple chat rooms, send formatted messages with image/file attachments, see typing indicators, know when other users are online, and receive push notifications for new messages. The project covers WebSocket architecture, event-driven programming, JWT authentication, file uploads with Multer, and message pagination.

Problem Statement

Real-time communication is fundamental to modern web applications - from Slack and Discord to customer support chat and collaborative tools. Building a reliable chat system requires understanding WebSocket protocols, managing concurrent connections, handling message ordering, dealing with reconnection logic, and ensuring message delivery guarantees. The challenge is building a system that handles: multiple simultaneous chat rooms, message history with infinite scroll, typing indicators that don't cause excessive events, online/offline presence detection, file/image sharing, and message delivery receipts (sent, delivered, read).

Real World Use Cases

  • Team Collaboration: A remote team uses the chat app with channels for different projects (#frontend, #backend, #design) and direct messaging for private conversations.
  • Customer Support: An e-commerce site integrates the chat for live customer support with agent assignment, chat queues, and pre-defined response templates.
  • Community Platform: A gaming community creates rooms for different games, with voice chat integration, mod-only channels, and automated moderation bots.

Industry Relevance

Real-time features are expected in modern web applications. Socket.io powers real-time features for millions of users at companies like Trello, Patreon, and Microsoft. Understanding WebSocket architecture is critical for building collaborative tools (Google Docs, Figma), live dashboards (Datadog, Grafana), gaming platforms, and financial trading interfaces. The event-driven patterns you learn transfer to message queues (RabbitMQ, Kafka), event sourcing architectures, and real-time data pipelines.

Architecture Overview

The system uses a client-server architecture with WebSocket as the primary communication channel. The Socket.io server runs alongside the Express HTTP server on the same port, sharing the authentication context. Connections are organized into namespaces (default namespace for general chat, admin namespace for moderation). Within each namespace, rooms group users by chat channel. The server maintains an in-memory map of connected users (socketId -> userId) for fast presence lookups, while persisting messages to MongoDB for history. File uploads are handled via Multer middleware and stored on Cloudinary with URLs embedded in message objects. Message delivery uses an acknowledgement pattern where the client sends an ACK after rendering, enabling read receipts. The reconnection system uses exponential backoff with jitter and rejoins all rooms on reconnection.

Development Roadmap

Phase 1: Server Setup & Socket.io Fundamentals

Initialize the Express server with Socket.io. Implement the connection/disconnection lifecycle. Create the basic event system: client emits 'chat message', server broadcasts to all clients in the room. Test with a simple HTML client that connects and sends messages.

Phase 2: Authentication & User Management

Implement user registration and login with JWT tokens. Add socket middleware to authenticate connections using the JWT token sent in the auth handshake. Store user profiles in MongoDB with avatars and display names. Implement guest/anon access for non-registered users.

Phase 3: Room Management

Implement room CRUD: create room, list rooms, join room, leave room, delete room (admin only). Messages are scoped to rooms - users only receive events for rooms they've joined. Room metadata includes name, description, topic, member count, and creation date.

Phase 4: Typing Indicators & Presence

Implement typing indicator events: 'typing:start' and 'typing:stop' with 2-second debounce. Broadcast only to the room. Implement online presence with socket connection/disconnection tracking. Add a heartbeat every 30 seconds to detect stale connections. Show online/away/offline status with colored indicators.

Phase 5: Message History & File Uploads

Store messages in MongoDB with fields: content, sender, room, timestamp, type (text/image/file), and metadata. Implement cursor-based pagination for infinite scroll. Add Multer middleware for file uploads with Cloudinary storage. Support image preview, file download, and drag-and-drop upload.

Implementation Guide

Step 1: Socket.io Server with Authentication Middleware

Set up the Socket.io server with JWT authentication middleware that validates tokens on connection and attaches user data to the socket object.

const express = require('express');
const http = require('http');
const { Server } = require('socket.io');
const jwt = require('jsonwebtoken');
const cors = require('cors');

const app = express();
app.use(cors());
const server = http.createServer(app);

const io = new Server(server, {
  cors: { origin: '*', methods: ['GET', 'POST'] },
  pingInterval: 25000,
  pingTimeout: 20000
});

// Authentication middleware for Socket.io
io.use((socket, next) => {
  const token = socket.handshake.auth.token;
  if (!token) {
    return next(new Error('Authentication required'));
  }
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    socket.userId = decoded.userId;
    socket.username = decoded.username;
    socket.avatar = decoded.avatar;
    next();
  } catch (err) {
    next(new Error('Invalid token'));
  }
});

const onlineUsers = new Map(); // userId -> Set

io.on('connection', (socket) => {
  console.log(`User connected: ${socket.username} (${socket.id})`);

  // Track online users
  if (!onlineUsers.has(socket.userId)) {
    onlineUsers.set(socket.userId, new Set());
  }
  onlineUsers.get(socket.userId).add(socket.id);
  io.emit('user:online', { userId: socket.userId, username: socket.username });

  socket.on('disconnect', () => {
    const userSockets = onlineUsers.get(socket.userId);
    if (userSockets) {
      userSockets.delete(socket.id);
      if (userSockets.size === 0) {
        onlineUsers.delete(socket.userId);
        io.emit('user:offline', { userId: socket.userId });
      }
    }
    console.log(`User disconnected: ${socket.username}`);
  });
});

server.listen(process.env.PORT || 3000, () => {
  console.log('Server running on port', process.env.PORT || 3000);
});

Step 2: Room Management Events

Implement room creation, joining, leaving, and message broadcasting scoped to rooms. Users can be in multiple rooms simultaneously.

// In-memory room store (use MongoDB for persistence)
const rooms = new Map();

// Add default rooms
rooms.set('general', { id: 'general', name: 'General', description: 'General discussion', createdBy: 'system', createdAt: new Date() });
rooms.set('random', { id: 'random', name: 'Random', description: 'Random topics and fun', createdBy: 'system', createdAt: new Date() });

socket.on('room:join', (roomId, callback) => {
  if (!rooms.has(roomId)) {
    return callback({ error: 'Room not found' });
  }
  socket.join(roomId);
  socket.currentRooms = socket.currentRooms || [];
  socket.currentRooms.push(roomId);
  const room = rooms.get(roomId);
  io.to(roomId).emit('user:joined', {
    userId: socket.userId,
    username: socket.username,
    roomId
  });
  callback({ success: true, room });
});

socket.on('room:leave', (roomId) => {
  socket.leave(roomId);
  socket.currentRooms = socket.currentRooms.filter(r => r !== roomId);
  io.to(roomId).emit('user:left', {
    userId: socket.userId,
    username: socket.username,
    roomId
  });
});

socket.on('room:create', (data, callback) => {
  const { name, description } = data;
  const id = crypto.randomBytes(8).toString('hex');
  const room = { id, name, description, createdBy: socket.userId, createdAt: new Date(), memberCount: 0 };
  rooms.set(id, room);
  socket.join(id);
  io.emit('room:created', room);
  callback({ success: true, room });
});

socket.on('room:list', (callback) => {
  const roomList = Array.from(rooms.values()).map(r => ({
    ...r,
    memberCount: io.sockets.adapter.rooms.get(r.id)?.size || 0
  }));
  callback(roomList);
});

Step 3: Message Handling with Persistence and ACKs

Handle message sending, storage, and delivery with read receipts. Messages are stored in MongoDB and broadcast to the room.

const Message = require('../models/Message');

socket.on('message:send', async (data, callback) => {
  try {
    const { roomId, content, type = 'text', metadata = {} } = data;

    if (!roomId || !content?.trim()) {
      return callback({ error: 'Message content is required' });
    }

    // Create and save message
    const message = new Message({
      roomId,
      sender: socket.userId,
      senderName: socket.username,
      senderAvatar: socket.avatar,
      content: content.trim(),
      type,
      metadata,
      timestamp: new Date()
    });
    await message.save();

    // Broadcast to room
    io.to(roomId).emit('message:new', message.toObject());

    // Acknowledge with message ID and timestamp
    callback({
      success: true,
      messageId: message._id,
      timestamp: message.timestamp
    });
  } catch (err) {
    callback({ error: 'Failed to save message' });
  }
});

// Message history with cursor-based pagination
socket.on('message:history', async (data, callback) => {
  try {
    const { roomId, cursor, limit = 50 } = data;
    const query = { roomId };
    if (cursor) {
      query.timestamp = { $lt: new Date(cursor) };
    }
    const messages = await Message.find(query)
      .sort({ timestamp: -1 })
      .limit(limit)
      .lean();
    callback({
      messages: messages.reverse(),
      hasMore: messages.length === limit,
      nextCursor: messages.length > 0 ? messages[0].timestamp : null
    });
  } catch (err) {
    callback({ error: 'Failed to load messages' });
  }
});

// Message read receipt
socket.on('message:read', async (data) => {
  const { messageId, roomId } = data;
  io.to(roomId).emit('message:read', {
    messageId,
    userId: socket.userId,
    readAt: new Date()
  });
});

Step 4: Typing Indicators with Throttling

Implement typing indicators with event throttling to prevent flooding the network with typing events. Users stop typing after 2 seconds of inactivity.

const typingUsers = new Map(); // roomId -> Map

socket.on('typing:start', (roomId) => {
  if (!roomId) return;
  socket.to(roomId).emit('typing:update', {
    userId: socket.userId,
    username: socket.username,
    isTyping: true,
    roomId
  });

  // Auto-stop typing after 3 seconds
  if (!typingUsers.has(roomId)) {
    typingUsers.set(roomId, new Map());
  }
  const roomTyping = typingUsers.get(roomId);
  if (roomTyping.has(socket.userId)) {
    clearTimeout(roomTyping.get(socket.userId));
  }
  roomTyping.set(socket.userId, setTimeout(() => {
    socket.to(roomId).emit('typing:update', {
      userId: socket.userId,
      username: socket.username,
      isTyping: false,
      roomId
    });
    roomTyping.delete(socket.userId);
    if (roomTyping.size === 0) typingUsers.delete(roomId);
  }, 3000));
});

socket.on('typing:stop', (roomId) => {
  if (!roomId) return;
  const roomTyping = typingUsers.get(roomId);
  if (roomTyping?.has(socket.userId)) {
    clearTimeout(roomTyping.get(socket.userId));
    roomTyping.delete(socket.userId);
    if (roomTyping.size === 0) typingUsers.delete(roomId);
  }
  socket.to(roomId).emit('typing:update', {
    userId: socket.userId,
    username: socket.username,
    isTyping: false,
    roomId
  });
});

Testing Guide

Connection Testing: Open multiple browser tabs and connect as different users. Verify each user sees the correct online status. Disconnect a tab and verify the offline event fires. Test reconnection by stopping and restarting the server.

Room Testing: Create a room from one client, verify all clients see the new room. Join multiple rooms and verify messages only appear in the correct room. Test that leaving a room stops receiving messages.

Message Testing: Send messages with special characters, emoji, and long text. Verify messages persist after page refresh. Test message pagination by sending 100+ messages. Send an empty message and verify it's rejected. Test file uploads with images, PDFs, and large files.

Typing Indicators: Open two tabs as different users. Type in one tab and verify the typing indicator appears in the other. Stop typing and verify the indicator disappears after 3 seconds. Test rapid typing and verify it doesn't cause multiple events.

Authentication Testing: Try connecting without a token (should be rejected). Use an expired token (should be rejected). Switch users and verify the correct identity is shown. Test that disconnected users are removed from online status.

Deployment Guide

Server Deployment: Deploy the Node.js server to Railway, Render, or Fly.io. These platforms support WebSocket connections natively. For traditional hosting (AWS EC2, DigitalOcean), ensure the load balancer supports WebSocket (sticky sessions or WebSocket passthrough). Set environment variables for JWT_SECRET, MONGODB_URI, and CLOUDINARY_URL.

Client Deployment: If using a separate client, deploy to Vercel or Netlify. Configure the client with the server URL via environment variable. For the vanilla HTML client, update the server URL in the script.

Scaling: For horizontal scaling, use Socket.io's Redis adapter to share events across multiple server instances. Set up a Redis server and configure the adapter: new RedisAdapter(io, pubClient, subClient). Use MongoDB replica sets for message persistence reliability.

Monitoring: Use Socket.io's built-in metrics (io.engine.clientsCount, adapter.rooms). Set up Prometheus + Grafana for monitoring. Configure alerts for high connection counts, message latency, and error rates.

Common Mistakes

1. Not Handling Socket.io Reconnection Properly
When a client reconnects after a temporary disconnect, they lose their room membership. Always listen for the 'connect' event on the client side and rejoin all previously joined rooms. Use the socket.io.reconnect() callback to restore state. Store the list of joined rooms in localStorage for recovery after page refresh.
2. Broadcasting Messages Without Server Validation
Never trust client-side data. Always validate message content, room membership, and user permissions on the server before broadcasting. A malicious client could emit forged events to impersonate other users or join restricted rooms. Validate the JWT token on every reconnection and check room membership before processing message:send events.
3. Memory Leaks from Event Listener Accumulation
Every time a component mounts and attaches a socket event listener, it creates a new listener. Without proper cleanup, these listeners accumulate and cause double-firing events and memory leaks. Always remove event listeners in the component cleanup function. Use socket.off() or socket.removeAllListeners() in the useEffect cleanup.
4. Excessive Typing Indicator Events
Without throttling, every keystroke emits a typing event, causing 10-30 events per second per user. With 100 users typing, this floods the network. Implement client-side throttling: emit typing:start at most once per second, and auto-emit typing:stop after 2 seconds of inactivity. On the server, track typing state per user per room and debounce the stop event.

Interview Questions

Q: Explain how WebSockets differ from HTTP and when you would use each.
A: HTTP is a request-response protocol where the client initiates communication. Each request opens a new connection (or reuses one via keep-alive). WebSocket provides a full-duplex persistent connection over a single TCP socket, allowing the server to push data to the client without polling. Use HTTP for REST APIs, CRUD operations, and cacheable content. Use WebSocket for real-time features: chat, live notifications, collaborative editing, gaming, financial tickers, and live dashboards. WebSocket has lower latency for bidirectional communication but requires more server resources per connection.
Q: How does Socket.io handle reconnection and what configuration options are available?
A: Socket.io client automatically attempts reconnection when the connection drops. Key options: reconnection (boolean, default true), reconnectionAttempts (Infinity), reconnectionDelay (1000ms initial), reconnectionDelayMax (5000ms), randomizationFactor (0.5, adds jitter). The client uses exponential backoff: delay doubles with each attempt up to reconnectionDelayMax, with jitter to prevent thundering herd. On reconnection, the client re-authenticates and re-emits any buffered events. The 'reconnect' event fires on successful reconnection.
Q: What is the difference between Socket.io namespaces and rooms?
A: Namespaces are logical channels that create separate communication contexts on the same server. Each namespace has its own event handlers, middleware, and connection pool. Think of namespaces like separate applications (e.g., /chat, /admin, /analytics). Rooms are sub-channels within a namespace for grouping connected sockets (e.g., different chat channels). A socket can be in multiple rooms but typically joins one namespace. Rooms are created dynamically when sockets join them and are removed when empty. Use namespaces for different features, rooms for sub-grouping within a feature.
Q: How do you scale Socket.io horizontally across multiple servers?
A: Use Socket.io's Redis adapter to share events across server instances. The adapter uses Redis Pub/Sub: when one server emits an event, it publishes to Redis, and all other servers receive the event and forward it to their connected clients. All clients connect to the same Redis instance (or cluster). For sticky sessions, configure the load balancer to route the same client to the same server (required for long-polling transport). Configure the adapter: const { createAdapter } = require('@socket.io/redis-adapter'); const pubClient = redis.createClient(); const subClient = pubClient.duplicate(); io.adapter(createAdapter(pubClient, subClient)).
Q: How would you implement end-to-end encryption in a chat application?
A: Implement the Signal Protocol (used by WhatsApp and Signal) for end-to-end encrypted messaging. Each user has a public/private key pair. Messages are encrypted with AES-256-GCM using a symmetric key that's exchanged using the Diffie-Hellman key exchange. Keys are stored client-side in IndexedDB. The server only sees encrypted ciphertext and cannot read messages. Implement Perfect Forward Secrecy using ephemeral keys that change with each session. Use a trusted key verification mechanism (QR code scanning or fingerprint comparison) to prevent man-in-the-middle attacks. The server handles key distribution, room membership, and delivery but never has access to plaintext.
