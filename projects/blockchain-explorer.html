Blockchain Explorer

Intermediate 45-60 hours 7 technologies Capstone

Project Overview

Build a comprehensive blockchain explorer that visualizes and analyzes blockchain data in real-time. This project covers block propagation, transaction tracking, smart contract analysis, network statistics, and wallet monitoring. You will build a full-stack application with a Node.js/Python backend that connects to Ethereum (via Web3.js/ethers.js), a React frontend with real-time data visualization, and a PostgreSQL database for indexing historical data. The explorer will support multiple networks (Ethereum mainnet, Goerli testnet, Polygon), provide advanced search capabilities, gas fee estimation, and token analytics. Deploy as a public-facing explorer similar to Etherscan but with customizable dashboards and alerting features.

Problem Statement

Blockchain networks generate massive amounts of on-chain data — blocks, transactions, logs, internal transactions, token transfers, and contract state changes. While public explorers like Etherscan exist, they offer limited customization, no programmatic access for power users, and no real-time alerting. Developers building on blockchain need: (1) Real-time transaction monitoring with custom filters, (2) Smart contract verification and interaction tools, (3) Gas price analytics for optimal transaction timing, (4) Portfolio tracking across multiple addresses and networks, (5) WebSocket-based push notifications for on-chain events. The goal is to build an extensible, self-hosted blockchain explorer that provides deep insights into blockchain data with a modern, responsive UI and a comprehensive REST + WebSocket API.

Real World Use Cases

DeFi Portfolio Tracker

Monitor wallet addresses in real-time for token transfers, DeFi protocol interactions (Uniswap swaps, Aave deposits), and yield farming positions. Alerts via webhook/email when significant transactions occur.

Smart Contract Auditor

Analyze contract bytecode, verify source code, trace internal transactions, and detect suspicious patterns (honeypots, rug pulls, hidden mint functions). Used by security auditors before launching contracts.

Gas Optimization Tool

Track historical gas prices across ETH, Polygon, and BSC. Predict optimal gas prices using ML models. Create automated transaction submission at predicted low-gas times. Save 20-30% on gas fees.

Network Health Monitor

Track validator activity, block propagation times, mempool size, uncle/orphan block rates, and chain reorganization events. Critical for infrastructure providers running nodes.

Data Flow: (1) Indexer connects to Ethereum via WebSocket JSON-RPC, subscribing to new block headers. (2) On each new block, indexer fetches full block + transaction data + receipts + logs. (3) Data is normalized, enriched (token transfer decoding, internal tx tracing), and stored in PostgreSQL. (4) Relevant data is also indexed in Elasticsearch for search. (5) Updated data is pushed to Redis pub/sub, which the API gateway subscribes to. (6) WebSocket connections relay real-time updates to connected frontend clients. (7) Frontend displays data with automatic re-rendering via React state subscriptions.

UI/UX Design Strategy

The explorer follows a data-dense design philosophy — showing maximum information at a glance without overwhelming the user:

  • Dashboard as Landing: The home page shows a live dashboard with: latest blocks (auto-scrolling list), gas price gauge with trend arrow, mempool size indicator, daily transaction count chart, and top tokens by transfer volume. All charts update in real-time via WebSocket.
  • Search-First Navigation: A prominent search bar accepts any input (address, tx hash, block number, domain name) and auto-detects the type. Results appear in a dropdown with instant preview. Use Elasticsearch's auto-complete with fuzzy matching.
  • Detail Pages with Tabbed Layout: Transaction detail shows tabs: Overview (value, gas, status), Logs (decoded event list), Internal Txs (call trace tree), and Transfers (tokens moved). Each tab is independently scrollable with copy buttons for important values.
  • Address View with Portfolio: Address pages show: ETH balance + USD value, token holdings with current prices, recent transactions (filterable by incoming/outgoing), internal transactions, and contract interactions. Whale addresses get special badges.
  • Color Coding: Transactions color-coded by status (success=green, failed=red, pending=yellow). Gas prices shown with heat colors (low=green, medium=yellow, high=red, extreme=purple). Token transfers show direction arrows (inward=blue, outward=red).
  • Responsive Data Tables: All lists use virtualized scrolling for performance. Columns are configurable and sortable. Clicking a row navigates to the detail page. Copy-to-clipboard buttons for hashes and addresses.

Development Roadmap

Phase 1: Core Indexer (Week 1-2)

Set up Node.js project with TypeScript. Implement WebSocket connection to Ethereum node via ethers.js. Build block listener that subscribes to 'block' events. Implement block processor that fetches full block data, transactions, and receipts. Store in PostgreSQL via Prisma ORM. Handle chain reorganizations (reorgs up to 6 blocks deep). Create Docker environment with postgres + redis.

Phase 2: Transaction Enrichment (Week 3-4)

Build transaction processor that analyzes each tx for: token transfers (decode ERC20 Transfer events), internal transactions (via debug_traceTransaction), contract creation detection, and log parsing. Implement ABI decoding for common standards (ERC20, ERC721, ERC1155, Uniswap, OpenSea). Create token transfer table and internal transaction table.

Phase 3: Search & Indexing (Week 5-6)

Set up Elasticsearch cluster (single node for dev). Index blocks, transactions, addresses, and tokens with appropriate mappings. Implement unified search endpoint that queries all indices and ranks results by relevance. Add auto-complete suggestions. Build fuzzy search for addresses (Levenshtein distance on last 4 chars).

Phase 4: API Gateway (Week 7-8)

Build Express.js REST API with all endpoints. Implement caching layer with Redis (cache blocks/txs for 12 seconds = one block time). Add rate limiting (100 req/min for free tier, 1000 for premium). Set up Socket.io with Redis adapter for horizontal scaling. Create WebSocket rooms for address monitoring.

Phase 5: Frontend Dashboard (Week 9-10)

Create React app with Vite. Build dashboard with real-time block list, gas tracker, and network stats. Implement search bar with auto-complete. Build block detail, transaction detail, and address view pages. Add token transfer tables and internal transaction tree views.

Phase 6: Smart Contracts & Verification (Week 11-12)

Implement contract verification system: accept source code + compiler version + settings, compile with Solidity compiler, compare bytecode against on-chain code. Store verified sources and ABIs. Build contract interaction page with read/write function calls using ethers.js.

Phase 7: Watch Lists & Alerts (Week 13-14)

Build user authentication (JWT). Create watched address CRUD with webhook notifications. Implement alert evaluation engine: check each new transaction against watch rules (min value, specific tokens, counterparty filtering). Send alerts via webhook, email (SendGrid), and WebSocket push.

Phase 8: Analytics & Polish (Week 15-16)

Build analytics pipeline with materialized views for: daily active addresses, TPS history, gas price distribution, top tokens by volume, whale movement tracking. Add TradingView charts for gas history. Implement portfolio tracker with USD value calculation. Load testing with k6. Production deployment with PM2 cluster mode.

Implementation Guide

Step 1: Blockchain Indexer Core

// indexer/src/listeners/blockListener.ts import { ethers } from 'ethers'; import { EventEmitter } from 'events'; import { BlockProcessor } from '../processors/blockProcessor'; import { logger } from '../utils/logger'; import { config } from '../config'; export class BlockListener extends EventEmitter { private provider: ethers.WebSocketProvider; private processor: BlockProcessor; private latestBlock: number = 0; private isSyncing: boolean = false; constructor() { super(); this.provider = new ethers.WebSocketProvider(config.RPC_WS_URL); this.processor = new BlockProcessor(this.provider); } async start() { // Subscribe to new block headers this.provider.on('block', async (blockNumber: number) => { logger.info(`New block detected: ${blockNumber}`); this.latestBlock = blockNumber; if (!this.isSyncing) { this.isSyncing = true; try { await this.processor.processBlock(blockNumber); this.emit('blockProcessed', blockNumber); } catch (error) { logger.error(`Failed to process block ${blockNumber}:`, error); // Queue for retry await this.queueForRetry(blockNumber); } finally { this.isSyncing = false; } } }); // Handle reconnections this.provider.on('error', (error) => { logger.error('WebSocket error:', error); this.reconnect(); }); } private async queueForRetry(blockNumber: number) { // Store failed blocks in Redis for retry await config.redis.lpush('failed:blocks', blockNumber.toString()); } private async reconnect() { logger.info('Attempting WebSocket reconnection...'); this.provider.removeAllListeners(); this.provider = new ethers.WebSocketProvider(config.RPC_WS_URL); // Backfill missed blocks const current = await this.provider.getBlockNumber(); for (let i = this.latestBlock + 1; i <= current; i++) { await this.processor.processBlock(i); } await this.start(); } async backfill(fromBlock: number, toBlock: number) { logger.info(`Backfilling blocks ${fromBlock} to ${toBlock}`); for (let i = fromBlock; i <= toBlock; i++) { await this.processor.processBlock(i); if (i % 100 === 0) logger.info(`Backfill progress: ${i - fromBlock}/${toBlock - fromBlock}`); } } }

Step 2: Block Processor

// indexer/src/processors/blockProcessor.ts import { ethers } from 'ethers'; import { prisma } from '../services/dbService'; import { searchService } from '../services/searchService'; import { cacheService } from '../services/cacheService'; import { TxProcessor } from './txProcessor'; import { logger } from '../utils/logger'; export class BlockProcessor { private provider: ethers.Provider; private txProcessor: TxProcessor; constructor(provider: ethers.Provider) { this.provider = provider; this.txProcessor = new TxProcessor(provider); } async processBlock(blockNumber: number) { const startTime = Date.now(); // Fetch full block with transactions const block = await this.provider.getBlock(blockNumber, true); if (!block) throw new Error(`Block ${blockNumber} not found`); // Check for reorg const existing = await prisma.block.findUnique({ where: { number: blockNumber } }); if (existing && existing.hash !== block.hash) { logger.warn(`Reorg detected at block ${blockNumber}: ${existing.hash} -> ${block.hash}`); await this.handleReorg(blockNumber); } // Upsert block await prisma.block.upsert({ where: { number: blockNumber }, update: { hash: block.hash, parentHash: block.parentHash, timestamp: new Date(block.timestamp * 1000), miner: block.miner, gasUsed: block.gasUsed.toString(), gasLimit: block.gasLimit.toString(), baseFee: block.baseFeePerGas?.toString(), size: block.size, txCount: block.transactions.length, difficulty: block.difficulty.toString(), nonce: block.nonce, }, create: { number: blockNumber, hash: block.hash, parentHash: block.parentHash, timestamp: new Date(block.timestamp * 1000), miner: block.miner, gasUsed: block.gasUsed.toString(), gasLimit: block.gasLimit.toString(), baseFee: block.baseFeePerGas?.toString(), size: block.size, txCount: block.transactions.length, difficulty: block.difficulty.toString(), extraData: block.extraData, stateRoot: block.stateRoot, receiptsRoot: block.receiptsRoot, nonce: block.nonce, }, }); // Process each transaction const receipts = await Promise.all( block.transactions.map(tx => this.provider.getTransactionReceipt(tx.hash)) ); for (let i = 0; i < block.transactions.length; i++) { const tx = block.transactions[i]; const receipt = receipts[i]; if (receipt) { await this.txProcessor.processTransaction(tx as ethers.TransactionResponse, receipt); } } // Index block in Elasticsearch await searchService.indexBlock(block); // Cache block await cacheService.set(`block:${blockNumber}`, block, 15); // Update network stats await this.updateNetworkStats(block); const duration = Date.now() - startTime; logger.info(`Block ${blockNumber} processed in ${duration}ms (${block.transactions.length} txs)`); // Publish to Redis for real-time updates await cacheService.publish('blocks:new', JSON.stringify({ number: blockNumber, hash: block.hash, txCount: block.transactions.length, timestamp: block.timestamp, })); } private async handleReorg(blockNumber: number) { // Delete block and all associated data await prisma.$transaction([ prisma.internalTransaction.deleteMany({ where: { blockNumber } }), prisma.tokenTransfer.deleteMany({ where: { blockNumber } }), prisma.log.deleteMany({ where: { blockNumber } }), prisma.transaction.deleteMany({ where: { blockNumber } }), prisma.block.delete({ where: { number: blockNumber } }), ]); // Notify frontend about reorg await cacheService.publish('system:reorg', JSON.stringify({ blockNumber })); } private async updateNetworkStats(block: ethers.Block) { const recentBlocks = await prisma.block.findMany({ orderBy: { number: 'desc' }, take: 100, }); if (recentBlocks.length >= 2) { const timeSpan = (recentBlocks[0].timestamp.getTime() - recentBlocks[recentBlocks.length - 1].timestamp.getTime()) / 1000; const avgBlockTime = timeSpan / (recentBlocks.length - 1); const totalGasUsed = recentBlocks.reduce((sum, b) => sum + BigInt(b.gasUsed), BigInt(0)); const avgGasPrice = totalGasUsed / BigInt(recentBlocks.length); await prisma.networkStat.create({ data: { timestamp: new Date(), blockNumber: block.number, tps: recentBlocks.reduce((sum, b) => sum + b.txCount, 0) / timeSpan, avgGasPrice: avgGasPrice.toString(), avgBlockTime, totalTxs24h: await this.getTxCount24h(), activeAddresses24h: await this.getActiveAddresses24h(), }, }); } } private async getTxCount24h(): Promise { const since = new Date(Date.now() - 24 * 60 * 60 * 1000); return prisma.transaction.count({ where: { createdAt: { gte: since } } }); } private async getActiveAddresses24h(): Promise { const since = new Date(Date.now() - 24 * 60 * 60 * 1000); const [fromCount, toCount] = await Promise.all([ prisma.transaction.groupBy({ by: ['from'], where: { createdAt: { gte: since } }, _count: true }), prisma.transaction.groupBy({ by: ['to'], where: { createdAt: { gte: since } }, _count: true }), ]); return new Set([...fromCount.map(r => r.from), ...toCount.map(r => r.to)]).size; } }

Blockchain Fundamentals

Block Structure: An Ethereum block contains: (1) Block header: parentHash, ommersHash, beneficiary (miner), stateRoot, transactionsRoot, receiptsRoot, logsBloom, difficulty, number, gasLimit, gasUsed, timestamp, extraData, mixHash, nonce, baseFeePerGas (EIP-1559). (2) Transaction list: array of signed transactions. (3) Uncle (ommer) headers: for PoW consensus.

Mining (PoW): Miners solve a computational puzzle by varying the nonce until the block hash is below the target difficulty. The difficulty adjusts every 1024 blocks to maintain ~13-second block time. With The Merge (Sep 2022), Ethereum switched to Proof of Stake — validators propose blocks based on their staked ETH.

Transaction Lifecycle: (1) User signs transaction with private key ? (2) Transaction broadcast to mempool ? (3) Validators/miners pick transactions based on priority fee ? (4) Transaction included in a block ? (5) Block propagated to network ? (6) After 12 confirmations (~2 minutes), transaction considered final.

Transaction Pools (Mempool)

The mempool contains all pending transactions that haven't been included in a block yet. Key implementation details:

  • Subscribe to 'pending' transactions via WebSocket JSON-RPC
  • Store pending txs in Redis with TTL of 5 minutes (or until mined)
  • Track: gas price, transaction age, sender nonce, contract interaction
  • Frontend shows real-time mempool with gas price sorting and search

Smart Contracts

Smart contracts are immutable programs deployed on the blockchain. Our explorer provides: (1) Source code verification (bytecode comparison), (2) ABI decoding for read/write function calls, (3) Event log parsing with named parameters, (4) Internal transaction tracing for contract-to-contract calls, (5) Security analysis: check for known vulnerability patterns (reentrancy, timestamp dependency, unchecked calls).

Web3 Integration

The explorer uses ethers.js v6 as the primary Web3 library. Key integration points: (1) WebSocket provider for real-time block/tx subscriptions, (2) JSON-RPC for historical data queries, (3) Contract class for interacting with verified contracts, (4) Wallet class for signing transactions (used in contract write interface), (5) EIP-1193 provider injection for MetaMask integration on the frontend.

Network Statistics

Real-time network statistics calculated every block: TPS (transactions per second over last 100 blocks), average block time, gas price distribution, active addresses per 24h, total transactions per 24h, uncle rate, average transaction value, top gas consumers (contracts), and MEV activity estimation. Displayed as live-updating charts on the dashboard.

Consensus Mechanisms

Proof of Work (PoW): Used by Ethereum pre-Merge. Miners compete to find a nonce such that hash(block_header) < target. Difficulty adjusts dynamically. Energy-intensive but proven secure.

Proof of Stake (PoS): Used by Ethereum post-Merge and Polygon. Validators stake 32 ETH to propose blocks. Slashing conditions penalize misbehavior. Energy-efficient (~99.95% less energy than PoW). Finality achieved after two epochs (~12.8 minutes) via Casper FFG.

Gas and Fee Estimation

EIP-1559 introduced a base fee (burned) + priority fee (tip to validator). Our gas estimator: (1) Tracks last 20 blocks base fees, (2) Predicts next block base fee using exponential moving average with 12.5% max increase, (3) Samples top 25% of pending tx priority fees, (4) Provides 4 tiers: safe (<30min wait), standard (<5min), fast (<1min), instant (next block), (5) Shows USD cost estimation using ETH/USD oracle price.

Deployment Guide

Docker Compose: Deploy indexer, API, PostgreSQL, Elasticsearch, Redis, and frontend using docker-compose. Use PM2 in cluster mode for the Node.js services. For production: (1) Use managed PostgreSQL (RDS), Elasticsearch (Elastic Cloud), and Redis (ElastiCache). (2) Deploy indexer on EC2 with auto-scaling. Security Considerations

  • API Key Authentication: All endpoints require valid API key in header. Keys are hashed with bcrypt before storage. Support key rotation with multiple active keys per user.
  • Rate Limiting: 100 requests/min per API key. Implement token bucket algorithm with Redis. Premium tier gets 1000 requests/min.
  • WebSocket Connection Limits: Max 50 concurrent connections per user. Max 100 address subscriptions per connection.
  • Input Validation: Strict validation on all blockchain addresses (checksum validation), transaction hashes, and block numbers. Sanitize search queries to prevent injection.

Performance Optimization

  • Materialized Views: Pre-compute network stats, daily aggregates, and token analytics using PostgreSQL materialized views refreshed every block.
  • Connection Pooling: Use PgBouncer for PostgreSQL. Max 20 connections per service instance. Reserve 5 connections for admin queries.
  • Elasticsearch Tuning: Use filter cache for frequent queries (address lookups). Enable searchable snapshots for cold data. Index templates with appropriate analyzers for address hashes.
  • Redis Caching: Cache blocks (15s TTL), gas prices (10s), address balances (30s), and popular search results (60s). Use Redis Cluster for sharding in production.

Scaling Considerations

  • Indexer Scaling: Multiple indexer instances, each responsible for different block ranges. Use Redis for distributed coordination (which instance handles which range). Leader election for backfill jobs.
  • Database Sharding: Partition blocks and transactions tables by month for query performance. Use Citus (PostgreSQL extension) for distributed sharding across nodes.
  • API Horizontal Scaling: Stateless API behind ALB. Redis-based sessions for WebSocket affinity. Socket.io with Redis adapter for cross-instance events.
  • Multi-Chain Support: Separate indexer instances per chain. Unified API gateway that routes based on chain_id parameter. Cross-chain search in Elasticsearch with chain_id filter.

Common Mistakes

1. Not handling chain reorganizations: Blocks can be reorged (typically 1-2 blocks deep, up to 6 for large reorgs). Always check parent hash consistency. Store block hash and validate before processing transactions.
2. Rate limiting RPC calls: Infura/Alchemy have strict rate limits. Implement request queuing with exponential backoff. Cache responses aggressively. Use batch JSON-RPC requests where possible.
3. Ignoring gas price fluctuations: Gas prices can spike 10x during NFT mints or market volatility. Always show gas prices in both Gwei and USD. Warn users when submitting transactions during high congestion.
4. Inefficient database queries: Blockchain data is append-only and grows unboundedly. Always use pagination with cursor-based (not offset-based) pagination. Archive old data to cold storage. Use TimescaleDB for time-series data.
5. Missing ABI decoding for contract interactions: Raw transaction input data is opaque hex. Always attempt ABI decoding using known contract ABIs (OpenZeppelin, Uniswap, etc.) and function signature databases (4byte.directory).

Interview Questions

Q1: Explain how Ethereum blocks are structured and what changes EIP-1559 introduced.
A: Every block has a header (parentHash, stateRoot, transactionsRoot, receiptsRoot, logsBloom, difficulty, number, gasLimit, gasUsed, timestamp, extraData, baseFeePerGas). EIP-1559 introduced a base fee that gets burned (reducing ETH supply) and a priority fee (tip to validator). Blocks now have a target gas size of 15M with a max of 30M. Base fee adjusts: if block >50% full, base fee increases by max 12.5%; if <50%, it decreases.
Q2: How would you design a system to detect chain reorganizations in real-time?
A: Store (block_number, block_hash) pairs in Redis with a TTL of 100 blocks. On each new block, check if parent_hash matches the stored hash of (block_number - 1). If mismatch, walk back ancestors until finding the common ancestor. Delete reorged blocks from DB, re-fetch from common ancestor. For deep reorgs (>6 blocks), pause processing and alert operators.
Q3: Compare WebSocket vs polling for blockchain data ingestion.
A: WebSocket (eth_subscribe): real-time, lower latency, fewer RPC calls, but requires reconnection handling and can miss events during disconnection. Polling: simpler, deterministic, easier to backfill, but higher latency (12s block time) and more RPC calls. Best practice: use WebSocket for live tracking with polling fallback for missed blocks during reconnection.
Q4: How do you decode ERC20 Transfer events from raw logs?
A: The Transfer event has signature `Transfer(address indexed from, address indexed to, uint256 value)`, generating topic0 = keccak256('Transfer(...)'). Topics[1] = from (indexed), topics[2] = to (indexed). Data field = hex-encoded uint256 value. Use ethers.js Interface to parse: `const iface = new ethers.Interface(['event Transfer(address indexed from, address indexed to, uint256 value)'])` then `iface.parseLog({ topics, data })`.
Q5: Design a gas price prediction system.
A: (1) Historical data: store last 24h gas prices in PostgreSQL. (2) Feature engineering: time of day, day of week, pending tx count, mempool congestion, upcoming NFT mints/events. (3) Model: LightGBM or XGBoost with rolling window features. (4) Prediction: output 4 percentiles for safe/standard/fast/instant. (5) Fallback: exponential moving average of last 20 blocks when model is unavailable. (6) Update: retrain model daily with new data.
Q6: How would you implement full-text search across millions of transactions and addresses?
A: Use Elasticsearch with optimized mappings: (1) Blocks: index number, hash, miner, timestamp. (2) Transactions: index hash (keyword), from, to, value, method_signature (first 4 bytes of input data). (3) Addresses: index address, label, tags. Use multi_match queries with field boosting (hash^3, from^2). Implement auto-complete using edge_ngram tokenizer on addresses. Use search_as_you_type for instant suggestions. Separate index per chain for multi-chain support.
Q7: Explain the difference between internal transactions and regular transactions.
A: Regular transactions are signed by an EOA (externally owned account) and appear in the block's transaction list. Internal transactions are calls made by contracts during execution — they're not signed and don't appear in the block. Examples: (1) Contract A calls Contract B's function, (2) ETH transfers during contract execution, (3) Contract creation via CREATE/CREATE2 opcode. Internal transactions are obtained via debug_traceTransaction (requires archival node) or by parsing execution traces.
Q8: How do you handle blockchain data storage costs?
A: (1) Tiered storage: hot data (last 7 days) on NVMe SSD, warm (7-90 days) on standard SSD, cold (>90 days) on S3 Glacier. (2) Compression: use PostgreSQL column compression for large TEXT fields (input_data, bytecode). (3) Partitioning: partition blocks and transactions by month with automatic partition creation. (4) Archival: move blocks older than 1 year to a separate archive database. (5) Sampling: for analytics, store aggregated data (hourly/daily) instead of raw transactions.
Q9: How would you add support for a new blockchain (e.g., Solana)?
A: (1) Create new RPC client adapter for Solana's JSON-RPC (different API, different data structures). (2) Implement Solana-specific block/transaction/token processors (SPL tokens instead of ERC20). (3) Create unified data model that maps Solana concepts to our generic schema (account ? address, instruction ? transaction, token program ? token transfer). (4) Add chain_id to all tables for multi-tenant storage. (5) New frontend components for Solana-specific features (stake accounts, vote instructions).
Q10: Describe the architecture of a production blockchain explorer handling 100+ TPS.
A: (1) Distributed indexer fleet: 3-5 instances behind load balancer, each assigned block ranges via Redis coordinator. (2) Write-ahead log (Kafka) for blockchain events before DB write. (3) PostgreSQL with Citus extension sharded by block_number range. (4) Elasticsearch cluster with 3 nodes for search. (5) Redis cluster for caching with read-replicas. (6) API layer: 10+ instances behind ALB with WebSocket stickiness. (7) CDN (CloudFront) for API responses with short TTL. (8) Monitoring: Prometheus + Grafana for indexer lag, DB query latency, RPC call success rates.
