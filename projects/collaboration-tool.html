Real-Time Collaboration Tool

Advanced 55-75 hours 8 technologies Capstone

Project Overview

Build a real-time collaborative editing platform similar to Google Docs or Notion, supporting simultaneous multi-user editing with conflict resolution, presence awareness, version history, and rich text formatting. This project implements both Operational Transform (OT) and Conflict-free Replicated Data Types (CRDT) for synchronization, WebSocket-based real-time communication, and a robust undo/redo system. The backend is built with Node.js + TypeScript using a custom WebSocket server (or Socket.io), Redis for pub/sub and state management, and PostgreSQL for document persistence. The frontend uses React with ProseMirror (or Slate.js) for rich text editing, and Canvas-based cursor rendering for multi-user presence. You will implement a custom sync engine, differential sync algorithm, and a scalable WebSocket room architecture supporting thousands of concurrent editors per document.

Problem Statement

Real-time collaboration introduces fundamental distributed systems challenges: (1) Multiple users editing the same document simultaneously creates conflicts that must be resolved without data loss. (2) Network latency (50-300ms) means users see stale states and make edits on diverged versions. (3) Cursor positions, selections, and scroll positions must be synchronized across users. (4) Undo/redo becomes exponentially complex — should Alice undoing affect Bob's edits? (5) Scaling from 2 to 1000 editors per document requires fundamentally different architectures. Existing solutions (Google Docs, Notion, Coda) are proprietary or rely on Google's OT implementation. The challenge is to build a production-grade collaboration engine that handles all these concerns with sub-100ms operation propagation, offline editing support, and conflict resolution that never loses user data.

Learning Outcomes

  • Operational Transform (OT): Deep understanding of OT algorithms — transform operations against each other to maintain consistency. Implement client-server OT with acknowledgement (OT-with-server-control) and understand why OT is hard to scale to multiple users.
  • CRDT (Conflict-free Replicated Data Types): Master CRDT theory and implementation — specifically RGA (Replicated Growable Array) for ordered sequences and LWW-Register for last-writer-wins semantics. Understand how CRDT achieves eventual consistency without a central server.
  • WebSocket Architecture: Design scalable WebSocket infrastructure with rooms, namespaces, sticky sessions, and Redis pub/sub for horizontal scaling. Handle backpressure, reconnection, and message ordering.
  • Presence Systems: Implement real-time presence — cursors (position, color, name), selections (ranges), online/offline status, and typing indicators. Use Canvas overlay for cursor rendering over the editing surface.
  • Version History & Diffing: Build a version tree with branching (like Git for documents). Implement efficient diff algorithms (Myers' diff, patience diff) and snapshot compression.
  • Conflict Resolution: Design conflict resolution strategies for different data types: text (OT/CRDT), rich text (formatting + text), objects (field-level merge), and lists (position-based merge).
  • Scalability: Design a system supporting 1000+ concurrent editors per document using operation batching, incremental snapshots, and connection pooling.

Data Flow: (1) User types ? ProseMirror captures local operation. (2) Operation sent to sync engine which applies OT transformation against pending concurrent operations. (3) Transformed operation sent via WebSocket to server. (4) Server receives operation, applies server-side OT, broadcasts to all other clients. (5) Other clients receive operation, transform against their pending local operations, and apply to their ProseMirror document. (6) Presence data (cursor position) is sent separately at lower priority (every 50ms vs operations which are sent immediately).

UI/UX Design Strategy

The collaboration tool's design focuses on making multiuser editing feel natural and uncluttered:

  • Cursor Presence: Each user gets a uniquely colored cursor with their name label. Cursors animate smoothly between positions using CSS transforms. Selection ranges shown as translucent colored highlights behind text. Cursor position updates at 20fps to balance bandwidth and smoothness.
  • Minimal Interference: Remote cursors are visible but non-interactive — they don't interfere with clicking, selecting, or typing. Cursors fade out after 5 seconds of inactivity. User list in the header shows active editors with colored dots.
  • Typing Indicators: When a user is typing, a small bubble appears in the margin showing their name and a typing animation. Prevents accidental interruption of someone who is actively typing.
  • Conflict Awareness: When a conflict occurs that can't be automatically resolved, affected text is highlighted with a yellow background and a merge tool UI appears. Users can choose "accept mine", "accept theirs", or "view diff".
  • Version History Timeline: A sidebar timeline shows all versions with author avatars, timestamps, and a visual diff preview. Users can click any version to see a side-by-side diff. Version names are auto-generated "Alice edited paragraph 3".
  • Offline Indicator: When connection drops, a subtle banner appears: "Working offline — changes will sync when reconnected." Offline edits are queued and applied with operational transform when reconnecting.

Step 1: OT Core — Operation Types & Transform

// server/src/sync/operation.ts export enum OperationType { Insert = 'insert', Delete = 'delete', Retain = 'retain', Format = 'format', } export interface InsertOp { type: OperationType.Insert; position: number; text: string; attributes?: Record; clientId: string; seq: number; } export interface DeleteOp { type: OperationType.Delete; position: number; length: number; clientId: string; seq: number; } export interface FormatOp { type: OperationType.Format; position: number; length: number; attributes: Record; clientId: string; seq: number; } export type Operation = InsertOp | DeleteOp | FormatOp; export interface OperationBatch { documentId: string; operations: Operation[]; baseVersion: number; clientId: string; }
// server/src/sync/transform.ts import { Operation, OperationType } from './operation'; /** * OT Function f(op1, op2) -> [op1', op2'] * Transforms op1 against op2 so they can be applied in either order */ export function transform(op1: Operation, op2: Operation): [Operation, Operation] { // TP1: If ops are from the same client, they're already ordered if (op1.clientId === op2.clientId) { return [op1, op2]; } // Insert vs Insert if (op1.type === OperationType.Insert && op2.type === OperationType.Insert) { return transformInsertInsert(op1, op2); } // Delete vs Delete if (op1.type === OperationType.Delete && op2.type === OperationType.Delete) { return transformDeleteDelete(op1, op2); } // Insert vs Delete if (op1.type === OperationType.Insert && op2.type === OperationType.Delete) { return transformInsertDelete(op1, op2); } if (op1.type === OperationType.Delete && op2.type === OperationType.Insert) { const [op2Prime, op1Prime] = transformInsertDelete(op2, op1); return [op1Prime, op2Prime]; } // For format operations, treat like retains return transformRetainRetain(op1, op2); } function transformInsertInsert(op1: InsertOp, op2: InsertOp): [InsertOp, InsertOp] { // If at same position, use clientId as tiebreaker if (op1.position === op2.position) { if (op1.clientId < op2.clientId) { // op1 goes first, op2 shifts right by op1 text length return [ op1, { ...op2, position: op2.position + op1.text.length }, ]; } else { // op2 goes first, op1 shifts right return [ { ...op1, position: op1.position + op2.text.length }, op2, ]; } } if (op1.position < op2.position) { // op1 is before op2, op2 shifts right return [ op1, { ...op2, position: op2.position + op1.text.length }, ]; } else { // op2 is before op1, op1 shifts right return [ { ...op1, position: op1.position + op2.text.length }, op2, ]; } } function transformDeleteDelete(op1: DeleteOp, op2: DeleteOp): [DeleteOp, DeleteOp] { const o1End = op1.position + op1.length; const o2End = op2.position + op2.length; // No overlap if (o1End <= op2.position) { // op1 is entirely before op2, no transform needed for op1 // op2 shifts left by op1.length return [ op1, { ...op2, position: op2.position - op1.length }, ]; } if (o2End <= op1.position) { // op2 is entirely before op1 return [ { ...op1, position: op1.position - op2.length }, op2, ]; } // Partial/complete overlap if (op1.position <= op2.position) { // op1 starts first or same const overlapLen = Math.min(o1End, o2End) - op2.position; const op1Len = op1.length - (op2.position - op1.position); const effectiveDelete1 = overlapLen; return [ { ...op1, length: op1Len - overlapLen }, { ...op2, position: op1.position, length: op2.length - effectiveDelete1 }, ]; } else { // op2 starts first const overlapLen = Math.min(o2End, o1End) - op1.position; return [ { ...op1, position: op2.position, length: op1.length - overlapLen }, { ...op2, length: op2.length - overlapLen }, ]; } } function transformInsertDelete(op1: InsertOp, op2: DeleteOp): [InsertOp, DeleteOp] { // Where is the insert relative to the delete? if (op1.position <= op2.position) { // Insert before delete, delete shifts right return [op1, { ...op2, position: op2.position + op1.text.length }]; } else if (op1.position >= op2.position + op2.length) { // Insert after delete, insert shifts left return [ { ...op1, position: op1.position - op2.length }, op2, ]; } else { // Insert inside delete range — tricky // The insert survives (we keep it), but the delete splits around it // We split the delete into two: before insert and after insert // For simplicity, we keep the delete for the part before the insert const beforeLen = op1.position - op2.position; return [ { ...op1, position: op2.position + beforeLen }, { ...op2, length: beforeLen }, ]; } } /** * Compose: apply op1 then op2 = compose(op1, op2) */ export function compose(a: Operation, b: Operation): Operation { // Simplified compose — in production, handle all combinations if (a.type === OperationType.Insert && b.type === OperationType.Delete) { if (b.position >= a.position && b.position + b.length <= a.position + a.text.length) { // Deleting some of what was just inserted const keepStart = b.position - a.position; const keepEnd = (a.position + a.text.length) - (b.position + b.length); const newText = a.text.slice(0, keepStart) + a.text.slice(a.text.length - keepEnd); return { ...a, text: newText, position: a.position + keepStart }; } } // Default: just apply both (b already accounts for a's effect) return b; }

Step 2: CRDT — Replicated Growable Array (RGA)

// server/src/sync/crdt/RGA.ts import { v7 as uuidv7 } from 'uuid'; interface RGANode { id: string; // Unique ID (UUID v7 for time-ordered) value: string | null; // null = tombstone (deleted) createdAt: number; // Lamport timestamp or hybrid logical clock creator: string; // Client ID that created this node next: string | null; // ID of next node in sequence } /** * RGA (Replicated Growable Array) * A CRDT for ordered sequences with eventual consistency. * Each character/operation is a node in a linked list. * Conflicts resolved by: insert position (parent), then creator ID, then timestamp. */ export class RGA { private nodes: Map = new Map(); private head: string | null = null; // ID of first node private clock: number = 0; constructor() { // Initialize with empty document head // In practice, the document is a sequence of RGA nodes } /** * Local insert: create a new node at the given position after the target node */ localInsert(afterId: string | null, value: string, creator: string): RGANode { this.clock++; const node: RGANode = { id: uuidv7(), value, createdAt: this.clock, creator, next: null, }; if (afterId === null) { // Insert at beginning node.next = this.head; this.head = node.id; } else { const afterNode = this.nodes.get(afterId); if (!afterNode) throw new Error(`Node ${afterId} not found`); node.next = afterNode.next; afterNode.next = node.id; } this.nodes.set(node.id, node); return node; } /** * Remote insert: merge an insert operation from another client * Conflict resolution: find the right position based on causal ordering */ remoteInsert(node: RGANode): void { this.clock = Math.max(this.clock, node.createdAt) + 1; if (!this.nodes.has(node.id)) { this.nodes.set(node.id, node); this.mergeInsert(node); } } private mergeInsert(newNode: RGANode): void { // Find the correct position by finding where this node would have been inserted // We need to walk from head and find the right spot based on causal history if (this.head === null) { this.head = newNode.id; newNode.next = null; return; } // If this node was inserted at position 0 (no predecessor) // Try to determine position from causal relationships let prev: string | null = null; let current = this.head; while (current !== null) { const currentNode = this.nodes.get(current)!; // Check if new node should go before current node if (this.shouldPrecede(newNode, currentNode)) { if (prev === null) { newNode.next = this.head; this.head = newNode.id; } else { const prevNode = this.nodes.get(prev)!; newNode.next = prevNode.next; prevNode.next = newNode.id; } return; } prev = current; current = currentNode.next; } // Append to end const lastNode = this.nodes.get(prev!)!; lastNode.next = newNode.id; newNode.next = null; } /** * Determine if node a should come before node b * Rules: (1) If a is a descendant of b's predecessor, a goes after b * (2) Otherwise, compare by creator ID, then timestamp */ private shouldPrecede(a: RGANode, b: RGANode): boolean { if (a.creator < b.creator) return true; if (a.creator > b.creator) return false; return a.createdAt < b.createdAt; } /** * Delete a node (mark as tombstone) */ localDelete(nodeId: string): boolean { const node = this.nodes.get(nodeId); if (!node || node.value === null) return false; node.value = null; // Tombstone return true; } /** * Get the full document text (skip tombstones) */ getText(): string { let result = ''; let current = this.head; while (current !== null) { const node = this.nodes.get(current)!; if (node.value !== null) { result += node.value; } current = node.next; } return result; } /** * Get all non-tombstone nodes as an array */ getNodes(): RGANode[] { const result: RGANode[] = []; let current = this.head; while (current !== null) { const node = this.nodes.get(current)!; if (node.value !== null) { result.push(node); } current = node.next; } return result; } /** * Serialize for network transfer */ toJSON() { return { head: this.head, nodes: Array.from(this.nodes.entries()).map(([id, node]) => ({ id, value: node.value, createdAt: node.createdAt, creator: node.creator, next: node.next, })), clock: this.clock, }; WebSocket Fundamentals

WebSocket provides full-duplex communication over a single TCP connection. For collaboration tools, WebSocket is superior to HTTP polling because: (1) Sub-50ms message latency vs 200-500ms for polling, (2) Server can push updates without client request, (3) Persistent connection reduces TLS handshake overhead, (4) Binary framing reduces overhead vs HTTP headers. Key considerations: (1) Socket.io provides automatic fallback to long-polling when WebSocket is blocked, (2) Use sticky sessions for horizontal scaling (or Redis adapter), (3) Heartbeat interval of 25s to detect stale connections, (4) Message ordering guarantees — Socket.io preserves order within a single transport, (5) Backpressure handling — if client is too slow, server should buffer or drop presence updates.

Operational Transform (OT) Concepts

OT is the foundational technology behind Google Docs and many collaborative editors. Core principles:

  • Operation: A unit of change (insert N chars at position P, delete M chars at position P). Operations are small, composable, and transformable.
  • Transform Property 1 (TP1): If op1 and op2 are concurrent, then applying op1 then op2' (op2 transformed against op1) produces the same result as applying op2 then op1'.
  • Transform Property 2 (TP2): If op1, op2, op3 are concurrent, transform(op1, transform(op2, op3)) = transform(transform(op1, op2), op3).
  • Client-Server Model: Client sends operation to server. Server transforms against concurrent ops from other clients, applies, and broadcasts. Client receives ack with transformed version of own op.
  • Limitations: OT becomes exponentially complex with multiple simultaneous editors. Each new concurrent op requires transformation against all others. This is why Google Docs limits to ~100 editors.

CRDT (Conflict-free Replicated Data Types)

CRDT provides eventual consistency without a central server. Each character is a unique node with a globally unique ID. When conflicts occur (same position), CRDT uses deterministic rules (timestamp, creator ID) to decide order — no transformation needed. Benefits over OT: (1) No central server required for correctness, (2) Naturally handles offline editing — just merge on reconnect, (3) Scales to thousands of concurrent editors, (4) Simpler implementation (no transform functions). Trade-offs: (1) Larger metadata overhead (unique IDs per character), (2) Tombstones for deleted elements consume memory, (3) Less mature ecosystem. Libraries: yjs (used by Roam Research, Obsidian), automerge (used byJupyter Notebooks).

Presence Systems

Presence tracking has different performance characteristics than document sync: (1) Cursor updates are ephemeral — if lost, the next update will correct position, (2) Can tolerate higher latency (50-100ms vs 10-50ms for ops), (3) Higher frequency (20fps cursor updates vs 1-5 ops/sec typing), (4) Use separate WebSocket channel to avoid head-of-line blocking. Implementation: Presence updates sent every 50ms on a dedicated channel. Server broadcasts to room without persistence. Client renders with requestAnimationFrame for smooth animation. Fade out after 10 seconds of inactivity.

Version History and Diffing

Version history is implemented as a directed acyclic graph (DAG) of document states. Each version stores: (1) Incremental operations since parent, (2) A full snapshot every 50 versions (checkpoint), (3) Author, timestamp, and auto-generated description. Diff between versions uses Myers' O(ND) diff algorithm. For rich text, we diff the underlying plain text and apply formatting overlays. The version tree supports branching (collaborators can diverge and merge) — displayed as a visual timeline with branch indicators.

Conflict Resolution Strategies

  • Text (OT/CRDT): Automatic — characters never conflict because each has a unique ID. Simultaneous inserts at same position are ordered deterministically.
  • Rich Text Formatting: Last-writer-wins for formatting attributes. If Alice bolds "hello" and Bob underlines "hello", both formats are applied.
  • Object Fields: Last-writer-wins per field. If Alice edits title and Bob edits body, both changes are preserved.
  • List Items: Position-based conflict resolution using fractional indexing (Figma's approach) or CRDT list.
  • Unresolvable: When automatic resolution isn't possible (e.g., incompatible changes to same data), show merge UI similar to Git conflict markers. Users see both versions and pick the desired outcome.

Scalability with WebSocket Rooms

Horizontal scaling for WebSocket connections: (1) Use Socket.io Redis adapter — broadcasts are published to Redis, each server instance subscribes and forwards to connected clients, (2) Each document is a room — operations are only sent to clients in that room, (3) For 1000+ editors per document, use hierarchical rooms: doc:123:section:A, doc:123:section:B — clients only sync the section they're viewing, (4) Load balancer with sticky sessions (WebSocket affinity), (5) Separate read-only replicas for viewers who don't need to edit.

Deployment Guide

Docker Compose: Deploy sync server (3+ instances), PostgreSQL (RDS), Redis (ElastiCache), and frontend (Vercel). Use AWS ECS with Fargate for the sync server. Load balancer with proxy protocol v2 for WebSocket support. Socket.io Redis adapter for cross-instance communication. Auto-scaling based on active connections (target: 1000 connections per instance). Monitor: (1) Operation latency P50/P95/P99, (2) Active connections per instance, (3) Redis pub/sub message rate, (4) Document state memory usage.

Common Mistakes

1. Not handling concurrent operations correctly: OT transform functions must maintain TP1/TP2 properties. Many implementations handle 80% of cases but fail on edge cases like overlapping deletes or insert+delete at same position. Write property-based tests using fast-check to discover edge cases.
2. Mixing up operation ordering: The sequence of operations matters critically. Always include baseVersion in operation messages. Use vector clocks or version vectors to detect concurrent operations. Never assume operations arrive in causal order.
3. Ignoring memory usage: CRDT tombstones and operation buffers grow unboundedly. Implement snapshot compression — periodically create a full document snapshot and discard operations before it. Use Redis with maxmemory-policy for ephemeral state.
4. Blocking the main thread: CRDT merge operations on large documents (10K+ nodes) can take 10-100ms. Run heavy merge operations in a Web Worker on the client and a worker thread on the server. Use requestIdleCallback for non-critical updates.
5. No offline support in MVP: Users expect to work offline. Implement an offline queue using IndexedDB that stores pending operations. On reconnect, send all queued ops through OT transformation. Handle conflicts that arise from divergent offline edits.

Interview Questions

Q1: Explain the difference between OT and CRDT for collaborative editing.
A: OT transforms operations against each other so they can be applied in any order, requiring a central server to coordinate. CRDT gives each character a unique ID, so concurrent inserts naturally converge without transformation. OT is more mature (Google Docs), has less metadata overhead, but is harder to scale. CRDT is simpler for offline/peer-to-peer, handles arbitrary number of users, but uses more memory (tombstones, IDs per character).
Q2: How does Google Docs handle real-time collaboration at scale?
A: Google Docs uses OT with a central server. Each client sends operations to Google's servers, which transform against concurrent ops from other clients. They use a custom binary protocol over WebSocket. For scaling, they: (1) Share nothing between server instances (state in memory per document), (2) Use consistent hashing to route a document to a specific server, (3) Spawn per-document processes for isolation, (4) Limit concurrent editors to ~100 for performance.
Q3: Design a system supporting 1000 simultaneous editors on one document.
A: (1) Use CRDT (yjs/automerge) instead of OT for better scalability. (2) Implement virtual document sections — only sync the section each user is viewing. (3) Hierarchical WebSocket rooms: doc:123:section:5. (4) Dedicated server per document with connection pooling. (5) Operation batching — coalesce multiple operations into one message. (6) Throttle presence updates to 10fps. (7) Use Redis Streams for persistent operation log with consumer groups.
Q4: How do you implement undo in a collaborative editor?
A: Undo is complex because "undoing" Alice's edit shouldn't undo Bob's subsequent edit. Approach: (1) Each user has a local undo stack of their own operations. (2) When user triggers undo, generate an "inverse operation" (insert becomes delete, delete becomes insert). (3) The inverse operation is sent through OT like any other operation. (4) However, if another user's concurrent edit overlaps, the inverse might not restore the original state. (5) Better approach: selective undo — show a list of the user's recent operations and let them choose which to revert. Google Docs uses a "version history" approach instead of traditional undo.
Q5: Explain how cursor synchronization works in a collaborative editor.
A: Each client broadcasts their cursor position (character offset, selection start/end) at ~20fps. Position is relative to the current document state. When remote operations are applied, the cursor position must be transformed accordingly — if Alice inserts 10 chars before Bob's cursor, Bob's cursor shifts right by 10. This transformation uses the same OT transform functions. Cursors are rendered as Canvas overlays positioned using ProseMirror's coordsAtPos() — which maps document position to screen coordinates even after scrolling.
