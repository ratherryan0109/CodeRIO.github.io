Weather Dashboard

Beginner 3-4 hours JavaScript, OpenWeatherMap API, Chart.js

A complete engineering guide to building a real-time weather dashboard with OpenWeatherMap API integration, geolocation, interactive charts, and localStorage favorites.

Project Overview

This weather dashboard project teaches you how to integrate third-party REST APIs, handle asynchronous JavaScript with fetch and async/await, process JSON responses, and present data visually using Chart.js. Users can search for any city, view current conditions, see a 5-day forecast with interactive charts, save favorite cities to localStorage, and use their device geolocation for instant local weather. The project covers error handling, loading states, and responsive design for a polished user experience.

Problem Statement

Weather data is essential for daily planning but accessing it typically requires visiting multiple websites or apps that are cluttered with ads and unnecessary features. Developers need to understand how to consume REST APIs, handle asynchronous data fetching, manage API keys securely, and present data in a clean, user-friendly dashboard. The challenge is building an app that handles API errors gracefully, works offline with cached data, and provides an intuitive UI for quick weather lookups.

Real World Use Cases

  • Travel Planning: A traveler checks weather conditions for their destination city before packing. They save multiple cities to quickly compare forecasts.
  • Event Coordination: An event planner monitors weather across multiple outdoor venue locations to make real-time backup decisions.
  • Agricultural Monitoring: A farmer tracks temperature, humidity, and precipitation forecasts for crop management decisions across different fields.

Industry Relevance

API integration is a core skill for modern web developers. The OpenWeatherMap API serves over 1 million developers and is used by companies like Microsoft and Airbnb for weather data. The concepts you learn - REST API consumption, async/await patterns, error handling, data visualization - are directly applicable to integrating any third-party service (payment gateways, social media APIs, mapping services). Chart.js is used by over 3 million projects on GitHub for data visualization.

Architecture Overview

The application follows a modular architecture with separate concerns: API service layer handles all HTTP requests with interceptors for error handling; state management stores current weather data, forecast data, favorites, and UI state; the UI layer renders components using template functions. The app uses a facade pattern where the main controller coordinates between services and UI. API responses are cached in a Map object to reduce redundant network calls. Geolocation is requested on first load with a fallback to the last searched city. Chart.js instances are managed through a singleton pattern to prevent memory leaks.

Implementation Guide

Step 1: API Service Layer

Create a dedicated API service module that encapsulates all HTTP requests to OpenWeatherMap. This separation of concerns makes the code testable and maintainable.

const API_KEY = 'YOUR_API_KEY';
const BASE_URL = 'https://api.openweathermap.org/data/2.5';
const GEO_URL = 'https://api.openweathermap.org/geo/1.0';

const WeatherAPI = {
  async getCurrentWeather(city) {
    try {
      const response = await fetch(
        `${BASE_URL}/weather?q=${city}&appid=${API_KEY}&units=metric`
      );
      if (!response.ok) {
        const errorData = await response.json();
        throw new Error(errorData.message || 'City not found');
      }
      return await response.json();
    } catch (error) {
      if (error.name === 'TypeError') {
        throw new Error('Network error. Please check your connection.');
      }
      throw error;
    }
  },

  async getForecast(city) {
    try {
      const response = await fetch(
        `${BASE_URL}/forecast?q=${city}&appid=${API_KEY}&units=metric`
      );
      if (!response.ok) throw new Error('Failed to fetch forecast');
      return await response.json();
    } catch (error) {
      throw error;
    }
  },

  async getWeatherByCoords(lat, lon) {
    try {
      const response = await fetch(
        `${BASE_URL}/weather?lat=${lat}&lon=${lon}&appid=${API_KEY}&units=metric`
      );
      if (!response.ok) throw new Error('Failed to fetch weather');
      return await response.json();
    } catch (error) {
      throw error;
    }
  },

  async searchCities(query) {
    try {
      const response = await fetch(
        `${GEO_URL}/direct?q=${query}&limit=5&appid=${API_KEY}`
      );
      if (!response.ok) throw new Error('Search failed');
      return await response.json();
    } catch (error) {
      throw error;
    }
  }
};

Step 2: Geolocation with Fallback

Request the user's location on page load. If denied, fall back to the last searched city from localStorage or a default city. Handle all geolocation error states.

function getUserLocation() {
  return new Promise((resolve, reject) => {
    if (!navigator.geolocation) {
      reject(new Error('Geolocation is not supported by this browser'));
      return;
    }
    navigator.geolocation.getCurrentPosition(
      (position) => {
        resolve({
          lat: position.coords.latitude,
          lon: position.coords.longitude
        });
      },
      (error) => {
        let message = 'Failed to get location';
        switch (error.code) {
          case error.PERMISSION_DENIED:
            message = 'Location access was denied by the user';
            break;
          case error.POSITION_UNAVAILABLE:
            message = 'Location information is unavailable';
            break;
          case error.TIMEOUT:
            message = 'The request to get user location timed out';
            break;
        }
        reject(new Error(message));
      },
      { enableHighAccuracy: true, timeout: 10000, maximumAge: 300000 }
    );
  });
}

async function initializeWeather() {
  showLoadingState();
  try {
    const lastCity = Storage.getLastCity();
    if (lastCity) {
      await loadWeatherData(lastCity);
    } else {
      try {
        const coords = await getUserLocation();
        const data = await WeatherAPI.getWeatherByCoords(coords.lat, coords.lon);
        await loadWeatherData(data.name);
      } catch {
        await loadWeatherData('London');
      }
    }
  } catch (error) {
    showError(error.message);
  } finally {
    hideLoadingState();
  }
}

Step 3: Interactive Charts with Chart.js

function createTemperatureChart(labels, temps, unit) {
  const ctx = document.getElementById('tempChart').getContext('2d');
  if (tempChart) tempChart.destroy();
  tempChart = new Chart(ctx, {
    type: 'line',
    data: {
      labels: labels,
      datasets: [{
        label: `Temperature (${unit === 'metric' ? '°C' : '°F'})`,
        data: temps,
        borderColor: '#6c63ff',
        backgroundColor: 'rgba(108,99,255,0.1)',
        borderWidth: 2,
        fill: true,
        tension: 0.4,
        pointRadius: 4,
        pointBackgroundColor: '#6c63ff',
        pointBorderColor: '#fff',
        pointBorderWidth: 2
      }]
    },
    options: {
      responsive: true,
      maintainAspectRatio: false,
      plugins: {
        legend: { display: true, position: 'top' },
        tooltip: {
          callbacks: {
            label: (ctx) => `${ctx.parsed.y}°${unit === 'metric' ? 'C' : 'F'}`
          }
        }
      },
      scales: {
        x: { grid: { display: false }, ticks: { maxRotation: 45 } },
        y: {
          grid: { color: 'rgba(255,255,255,0.05)' },
          ticks: {
            callback: (v) => `${v}°${unit === 'metric' ? 'C' : 'F'}`
          }
        }
      },
      animation: {
        duration: 1000,
        easing: 'easeInOutQuart'
      }
    }
  });
}

Step 4: Favorites with localStorage CRUD

Implement full CRUD operations for favorite cities in localStorage. Render favorites as removable chips with click-to-load functionality.

const Storage = {
  getFavorites() {
    try {
      return JSON.parse(localStorage.getItem('weather-favorites')) || [];
    } catch {
      return [];
    }
  },

  addFavorite(city) {
    const favorites = this.getFavorites();
    const normalized = city.trim().toLowerCase();
    if (!favorites.some(f => f.toLowerCase() === normalized)) {
      if (favorites.length >= 10) {
        showToast('Maximum 10 favorites allowed', 'warning');
        return false;
      }
      favorites.push(city.trim());
      localStorage.setItem('weather-favorites', JSON.stringify(favorites));
      return true;
    }
    return false;
  },

  removeFavorite(city) {
    const favorites = this.getFavorites();
    const filtered = favorites.filter(f => f.toLowerCase() !== city.toLowerCase());
    localStorage.setItem('weather-favorites', JSON.stringify(filtered));
    return filtered;
  },

  getLastCity() {
    return localStorage.getItem('weather-last-city') || null;
  },

  setLastCity(city) {
    localStorage.setItem('weather-last-city', city);
  },

  getUnit() {
    return localStorage.getItem('weather-unit') || 'metric';
  },

  setUnit(unit) {
    localStorage.setItem('weather-unit', unit);
  }
};

Step 5: Debounced Search with Autocomplete

Implement a search input with debouncing to reduce API calls. Show autocomplete suggestions from the Geocoding API with a dropdown menu.

function debounce(fn, delay = 500) {
  let timeoutId;
  return function (...args) {
    clearTimeout(timeoutId);
    timeoutId = setTimeout(() => fn.apply(this, args), delay);
  };
}

const searchInput = document.getElementById('search-input');
const suggestionsContainer = document.getElementById('suggestions');

const handleSearch = debounce(async (query) => {
  if (query.length < 2) {
    suggestionsContainer.classList.remove('active');
    return;
  }
  try {
    const cities = await WeatherAPI.searchCities(query);
    renderSuggestions(cities);
  } catch (error) {
    console.error('Search failed:', error);
  }
}, 400);

searchInput.addEventListener('input', (e) => {
  handleSearch(e.target.value);
});

searchInput.addEventListener('keydown', (e) => {
  if (e.key === 'Enter') {
    e.preventDefault();
    const query = e.target.value.trim();
    if (query) loadWeatherData(query);
    suggestionsContainer.classList.remove('active');
  }
});

document.addEventListener('click', (e) => {
  if (!e.target.closest('.search-wrapper')) {
    suggestionsContainer.classList.remove('active');
  }
});

function renderSuggestions(cities) {
  suggestionsContainer.innerHTML = cities.map(city => `
    

      
      ${city.name}, ${city.state || ''} ${city.country}
    

  `).join('');
  suggestionsContainer.classList.add('active');
  suggestionsContainer.querySelectorAll('.suggestion-item').forEach(item => {
    item.addEventListener('click', () => {
      loadWeatherByCoords(
        parseFloat(item.dataset.lat),
        parseFloat(item.dataset.lon)
      );
      suggestionsContainer.classList.remove('active');
      searchInput.value = item.querySelector('span').textContent.split(',')[0];
    });
  });
}

Testing Guide

API Testing: Test with valid city names ("London", "Tokyo", "New York"), invalid names ("Xyzabc"), empty input, and special characters. Verify error messages are user-friendly. Test API key expiry by using an invalid key. Use browser network throttling to test slow connections.

Geolocation Testing: Test with location enabled, denied, and unavailable. Use Chrome DevTools Sensors panel to simulate different locations. Verify the fallback mechanism works when geolocation fails.

localStorage Testing: Add 10+ favorites to verify the limit. Remove all favorites and verify the empty state. Clear localStorage and confirm the app initializes with a default city. Test across browser private/incognito modes.

Chart Testing: Verify charts render with data, handle empty data arrays gracefully, and resize correctly on window resize. Test unit toggle (C/F) and confirm chart labels and tooltips update.

Deployment Guide

API Key Security: Never commit API keys to GitHub. Use a .env file locally and configure environment variables on the hosting platform. For static sites, use a serverless proxy (Netlify Functions, Vercel Edge Functions) to forward API requests with the key stored on the server.

Netlify Deployment: Create a netlify/functions/proxy.js file that forwards requests to OpenWeatherMap. Set the API key as a Netlify environment variable. Deploy by connecting your GitHub repository.

Vercel Deployment: Create an api/proxy.js serverless function. Use @vercel/node runtime. Set the API key in Vercel project settings under Environment Variables. Deploy with vercel CLI or GitHub integration.

GitHub Pages: For GitHub Pages, use a CORS proxy service like cors-anywhere or create a simple Cloudflare Worker that adds your API key to requests. Note that exposing the API key in client-side code is a security risk; always use a proxy.

Common Mistakes

1. Exposing API Keys in Client-Side Code
Hardcoding your OpenWeatherMap API key in JavaScript files makes it visible to anyone who views your site's source code. Always use a serverless proxy function or environment variables. If using a free API tier, restrict the API key to specific domains in the OpenWeatherMap dashboard.
2. Not Handling API Rate Limits and Errors
Free OpenWeatherMap API plans have rate limits (60 calls/minute). Without error handling, exceeding this limit causes silent failures. Implement retry logic with exponential backoff, show user-friendly error messages, and cache responses aggressively to minimize API calls.
3. Memory Leaks from Chart Instances
Creating new Chart.js instances without destroying previous ones causes memory leaks and duplicate charts. Always call chart.destroy() before creating a new chart instance. Use a single chart manager that tracks active instances and cleans up on component unmount or data refresh.
4. Incorrect Temperature Unit Conversion
OpenWeatherMap returns temperatures in Kelvin by default. Many developers forget to specify units=metric or units=imperial in their API requests, or they do the conversion incorrectly (converting from Kelvin instead of Celsius). Always use the API's built-in unit parameter to avoid manual conversion errors.

Interview Questions

Q: Explain the difference between Promise.then() and async/await. When would you use each?
A: Both handle asynchronous operations but differ in syntax. .then() chains are better for simple sequential operations and parallel execution with Promise.all(). async/await provides a synchronous-looking syntax that's more readable for complex logic with multiple dependent async calls, error handling with try/catch, and loops. In practice, async/await is preferred for most modern code because it reduces nesting and makes error handling more intuitive.
Q: How would you implement caching for API responses to reduce network calls?
A: Use a Map or WeakMap to store responses keyed by the request URL + parameters. Set a TTL (time-to-live) for each cache entry (e.g., 10 minutes for weather data). Check the cache before making a network request. Implement a cache eviction strategy (LRU or time-based). For more advanced caching, use the Cache API in service workers for offline support. Example: cache.set(city, { data, timestamp: Date.now() }) and check if (Date.now() - cached.timestamp < TTL) before returning cached data.
Q: How do you handle CORS errors when making API requests from the browser?
A: CORS errors occur when the API server doesn't include the appropriate Access-Control-Allow-Origin headers. Solutions: Use a proxy server (your own backend or a CORS proxy like cors-anywhere), use JSONP if the API supports it (limited to GET requests), or use serverless functions (Netlify Functions, Vercel Edge) that make the request server-side where CORS doesn't apply. For OpenWeatherMap, CORS is enabled by default for all origins.
Q: What is the difference between localStorage and sessionStorage? When would you use each?
A: localStorage persists data indefinitely with no expiration, even after the browser is closed. sessionStorage persists data only for the duration of the page session and is cleared when the tab is closed. Use localStorage for user preferences (favorite cities, theme, temperature unit). Use sessionStorage for temporary state like the last searched city during a browsing session, or for caching API responses that don't need to persist across sessions.
Q: How do you make a third-party API call from a static site without exposing your API key?
A: Use a serverless function as a proxy. On Netlify: create netlify/functions/weather.js that reads the API key from environment variables and forwards the request. On Vercel: create api/weather.js. On Cloudflare: create a Worker that adds the API key. The client makes requests to your own domain (e.g., /api/weather?city=London) and the serverless function adds the API key before forwarding to the third-party API. This keeps the key server-side and allows you to set domain restrictions.
