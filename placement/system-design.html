System Design

Architect scalable, reliable, and maintainable systems. Essential for senior engineering and architect-level interviews.

8 Modules 15+ Questions High Priority

Scalability

Horizontal scaling (add more machines) vs vertical scaling (upgrade hardware). Load balancers distribute traffic. Stateless design enables horizontal scaling.

CAP Theorem

Consistency, Availability, Partition Tolerance — pick two. CA (traditional RDBMS), CP (HBase, MongoDB), AP (Cassandra, DynamoDB). Network partitions are inevitable.

CP > AP > CA in distributed
systems (P is mandatory)

Load Balancing

Distributes requests across servers. Algorithms: round robin, least connections, IP hash, consistent hashing. Health checks ensure only healthy servers receive traffic.

Server3] healthy: all 3

Caching & CDN

Redis/Memcached for in-memory cache. CDN (CloudFront, CloudFlare) for static assets. Cache strategies: write-through, write-around, write-back. Cache invalidation is hard.

Cache miss: query DB + populate

Design Patterns for System Design

Microservices Architecture

Decompose monolith into small, independent services. Each owns its data, communicates via APIs/message queues. Benefits: independent deploy, scale, tech diversity. Challenges: distributed complexity, data consistency.

Event-Driven Architecture

Components communicate via events. Producers emit events; consumers react. Message brokers: Kafka, RabbitMQ, SQS. Enables loose coupling and async processing.

CQRS (Command Query Responsibility Segregation)

Separates read and write operations. Different models for commands (writes) and queries (reads). Often paired with Event Sourcing. Optimizes for different access patterns.

Database Sharding

Horizontal partitioning of data across multiple databases. Shard key determines data distribution. Challenges: cross-shard joins, resharding, hotspotting.

Common Interview Questions

Design a URL shortener (like bit.ly).

Key components: encoding (base62), hash collision handling, redirect (301), analytics. Scale: distributed ID generation, caching, NoSQL for mapping, CDN for redirects.

Design a chat system (like WhatsApp).

WebSocket for real-time, message queue for delivery, database for history, push notifications. Challenges: ordering, delivery guarantees, multi-device sync, end-to-end encryption.

Design a recommendation system.

Collaborative filtering (user-item similarity), content-based (item features), hybrid approaches. Offline batch processing + online real-time serving. Use matrix factorization, embeddings.

Design a rate limiter.

Token bucket, leaky bucket, sliding window log/counter. Distributed rate limiting requires Redis or similar. Return 429 Too Many Requests when limit exceeded.

Design a distributed key-value store.

Consistent hashing for partitioning, replication for fault tolerance, quorum for consistency (R+W > N). Gossip protocol for membership. Vector clocks for conflict resolution.

Resources & Best Practices

System Design Interview Framework

Step 1: Understand scope & requirements. Step 2: Estimate scale (DAU, QPS, storage). Step 3: Design high-level components. Step 4: Deep dive into key components. Step 5: Identify bottlenecks and optimize.

Key Tradeoffs to Discuss

Consistency vs availability (CAP), SQL vs NoSQL, synchronous vs async, batch vs stream, monolithic vs microservices, strong vs eventual consistency.

Must-Read Resources

Designing Data-Intensive Applications (DDIA), System Design Interview (Alex Xu), High Scalability blog, Awesome Scalability GitHub, Papers: Google Spanner, Dynamo, MapReduce, Bigtable.

Back-of-the-Envelope Numbers

QPS: peak ~2x average. Storage: think in TB/PB. Network: 1 Gbps ~ 125 MB/s. DB: MySQL ~1k QPS, Redis ~100k QPS. Cache: 1ms vs DB: 10-50ms. 80/20 rule for caching.
